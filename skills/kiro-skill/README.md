# Kiro 工作流 Skill

這是一個完整的系統化開發 Kiro Spec Driven Development Agent Skill for OpenCode，將功能想法系統化地轉變為高品質的程式碼實現。

## Skill 結構

```
kiro-skill/
├── SKILL.md                    # 主文件：核心工作流程和全局約束
├── phases/                     # 階段提示詞文件夾
│   ├── 01-requirements.md      # 第一階段：需求明確
│   ├── 02-design.md            # 第二階段：架構設計
│   ├── 03-tasks.md             # 第三階段：任務規劃
│   ├── 04-implementation.md    # 第四階段：程式碼實現
│   ├── 05-review.md            # 第五階段：程式碼複核
│   └── 06-optimization.md      # 第六階段：程式碼優化
└── README.md                   # 本文件：使用說明
```

## 何時使用這個 Skill

當你需要：

- 開發任何語言和框架的功能
- 將功能想法轉變為程式碼實現
- 進行需求梳理
- 架構設計
- 任務規劃
- 程式碼實現
- 程式碼複核
- 程式碼優化

OpenCode 會自動識別並使用這個 Skill。

## 六個階段工作流

1. **需求明確** - 將模糊想法轉化為清晰的需求文件
2. **架構設計** - 基於需求創建技術實現方案
3. **任務規劃** - 將設計分解為具體的編碼任務
4. **程式碼實現** - 按照任務清單逐步實現功能
5. **程式碼複核** - 嚴格複核程式碼品質、規範一致性和需求符合度
6. **程式碼優化** - 基於複核報告進行迭代改進，確保程式碼品質

每個階段都有詳細的實施指南保存在 `phases/` 目錄中。

## 全局約束

- ✅ 必須嚴格遵循六個階段，從需求明確開始，不能跳過
- ✅ 使用 thinkharder 模式深度思考
- ✅ 必須完全了解專案程式碼和規範
- ✅ 語言：繁體中文
- ✅ 嚴格遵守 workspace `AGENTS.md` 與 `{.agents, .kiro, or .claude}/steering/` active root 規則（本專案目前以 `.agents` 為主）
- ✅ 執行任務前建立 To-dos，精確到每個子任務
- ✅ 不自動執行 SQL 語句
- ✅ 從全局到細節的方法論
- ✅ 每個階段需要用戶明確確認

## 工作流程文件

每個功能在 `{.agents, .kiro, or .claude}/specs/{功能名}/` 的 active root 目錄下會生成以下文件（本專案目前使用 `.agents`）：

```
{.agents, .kiro, or .claude}/specs/{功能名}/
├── requirements.md      # 第一階段：需求文件
├── design.md           # 第二階段：設計文件
├── tasks.md            # 第三階段：任務清單
├── completed_tasks.md  # 第四階段：完成狀態記錄
├── review.md           # 第五階段：程式碼複核報告
└── optimization_log.md # 第六階段：優化記錄
```

## 階段文件說明

### phases/ 目錄

包含六個階段的詳細實施指南：

- **01-requirements.md** - 詳細說明如何收集和明確需求，包括 EARS 格式的驗收標準
- **02-design.md** - 詳細說明如何進行技術設計，包括架構、組件、資料模型等
- **03-tasks.md** - 詳細說明如何拆解任務，包括任務格式和避免的非編碼任務
- **04-implementation.md** - 詳細說明如何實現程式碼，包括實施策略和品質控制
- **05-review.md** - 詳細說明如何複核程式碼品質，包括多維度評估和評分機制
- **06-optimization.md** - 詳細說明如何優化程式碼，包括優化循環和終止條件

## 程式碼複核評分機制

### 評分維度

程式碼複核基於四個維度進行評估，每個維度 0-10 分：

- **需求符合度** (25% 權重) - 檢查每個需求是否完全實作
- **設計一致性** (25% 權重) - 驗證是否遵循設計文件規範
- **程式碼規範性** (25% 權重) - 檢查是否符合編碼規範
- **程式碼品質** (25% 權重) - 評估可讀性、可維護性、效能等

### 綜合得分標準

- **≥ 8.5 分**：程式碼品質優秀，可直接完成
- **6.0-8.4 分**：存在改進空間，建議優化
- **< 6.0 分**：必須進行優化

## 優化循環機制

- **第 1 輪優化**：全面修復 review.md 中識別的問題
- **第 2 輪優化**：針對剩餘問題進行深度優化
- **第 3 輪優化**：最終精修和完善（最多進行到此輪）

### 終止條件

- 綜合得分達到 8.5 分以上
- 或已完成 3 輪優化（無論得分如何）
- 或用戶決定停止優化

## 交互方式

Agent 會用自然、友好的方式與你協作：

- "需求看起來完整了，我們可以開始架構設計了嗎？"
- "設計方案感覺怎麼樣？如果認可的話，我可以開始拆解具體任務"
- "這個任務完成了，想要繼續下一個，還是先看看現在的成果？"
- "複核評分 X.X 分，要不要開始優化提升程式碼品質？"

## 注意事項

1. **每個階段都需要你的明確認可才能進入下一階段**
2. **程式碼實現階段一次只專注一個任務，不會自動繼續**
3. **SQL 腳本需要你手動執行**
4. **大任務會分解成小任務，採用從全局到細節的方法**
5. **程式碼複核和優化確保最終程式碼品質達到優秀標準**

## 協作原則

1. **順序推進** - 確保每個階段都有堅實的基礎
2. **用戶主導** - 始終等待你的認可才進入下一階段
3. **專注執行** - 實現階段一次只專注一個任務
4. **保持上下文** - 始終基於完整的前序文件進行工作
5. **持續改進** - 支持在任何階段返回修改和優化
6. **品質把關** - 每個階段都有明確的完成標準

## 與 kiro-laravel-skill 的區別

- **kiro-skill**：通用的系統化開發工作流，適用於任何語言和框架
- **kiro-laravel-skill**：專門針對 Laravel 後端開發，包含 Laravel 特定的設計標準和程式碼示例

如果你在開發 Laravel 後端功能，推薦使用 **kiro-laravel-skill**。
如果你在開發其他語言或框架的功能，使用 **kiro-workflow-skill**。

## 優化說明

此 Skill 的結構經過優化：

- **主文件精簡** - SKILL.md 保留核心和必要的提示詞
- **分階段管理** - 每個階段的詳細提示詞獨立維護在 phases/ 目錄
- **結構清晰** - 易於維護和更新，每個文件職責明確
- **漸進式披露** - Agent 只在需要時讀取詳細的階段指南

通過這種自然、系統的協作方式，確保每個功能都經過深思熟慮，從想法到程式碼的每一步都穩扎穩打，最終交付出色的產品。
