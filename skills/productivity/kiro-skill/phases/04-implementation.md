# 第四階段：程式碼實作

## 觸發條件

- 任務清單已完成並獲得您的認可
- 或您指定要實作某個具體任務

## 我的工作

### 1. 實施前準備

- 調整到 thinkharder 模式
- 必須完全了解專案程式碼，了解專案規範和具體實現，收集足夠完整的上下文，再開始工作
- 在執行任何任務之前，始終確保已閱讀 active root 下的 `{.agents, .kiro, or .claude}/specs/{功能名}/` 規格 requirements.md、design.md 和 tasks.md 文件；本專案目前以 `.agents` 為主
- 在沒有需求或設計的情況下執行任務將導致不準確的實作

### 2. 思考過程展示（每個任務必須）

在開始任何任務之前，需要清晰展示：

1. **問題分析** - 這個任務要解決什麼問題？
2. **方案選擇依據** - 為什麼選擇這種實作方式？有哪些備選方案？
3. **規範依據** - 參考了哪些規範規則？規範中的具體條款是什麼？
4. **參考程式碼** - 參考了哪些現有程式碼？為什麼參考這些程式碼？

示例輸出格式：

```markdown
任務 2.3 Service 層開發

【思考過程】
- 問題分析：需要實作 xxx 業務邏輯，涉及到 yyy 的處理
- 方案選擇依據：參考現有的 ZzzService 實作模式，因為業務場景類似
- 規範依據：根據 workspace `AGENTS.md` 或 active root 下 `{.agents, .kiro, or .claude}/steering/service-layer.mdc` 規範，Service 層應該...
- 參考程式碼：參考 `app/Services/ZzzService.php` 的實作方式

工作內容：創建 XxxService，實作核心業務邏輯

具體步驟：
1. 在 app/Services 目錄下創建 XxxService.php
2. 注入 XxxRepository 依賴
3. 實作 xxx 方法
4. 實作 yyy 方法
```

### 3. 程式碼分層規範複習（重要）

**在每個程式碼開發任務開始前，必須執行以下規範複習流程：**

#### 3.1 複習流程

對於涉及程式碼編寫的任務，必須：

1. **識別程式碼分層** - 明確當前任務涉及哪些程式碼分層（Controller、Service、Repository、Model、Enum 等）
2. **讀取對應規範** - 讀取 workspace `AGENTS.md` 與 active root 下 `{.agents, .kiro, or .claude}/steering/` 目錄對應分層的規範文件
3. **讀取參考文件** - 如果規範文件中指定了參考文件，**必須重新讀取這些參考文件**
4. **展示規範要點** - 向用戶展示該分層的核心規範要點
5. **確認遵循** - 在開發過程中嚴格遵循這些規範

#### 3.2 規範複習輸出格式

在開始程式碼開發任務前，必須輸出以下內容：

```markdown
【規範複習】

涉及程式碼分層：Service 層

已讀取規範文件：
- `{.agents, .kiro, or .claude}/steering/service-layer.mdc`（active root；本專案目前以 `.agents` 為主）

規範指定的參考文件（已重新讀取）：
- `app/Services/ExampleService.php`
- `app/Services/AnotherService.php`

核心規範要點：
1. Service 層負責業務邏輯處理
2. 必須注入 Repository 依賴
3. 方法命名規範：xxx
4. ...

【開始開發】
```

#### 3.3 多分層任務處理

如果一個任務涉及多個程式碼分層，必須：

- 逐個複習每個分層的規範
- 展示每個分層的核心要點
- 按業務模組的鏈路順序開發

#### 3.4 規範衝突處理

如果發現規範之間存在衝突或不明確：

- 暫停開發
- 向用戶說明衝突點
- 獲得用戶確認後繼續

### 4. 任務執行策略

- 查看任務清單中的任務詳情
- 如果請求的任務有子任務，總是從子任務開始
- **嚴格一次只專注一個任務**——不實作其他任務的功能
- 根據任務或其詳情中指定的任何需求驗證實作
- 任務完成後，需要在 active root 下的 `{.agents, .kiro, or .claude}/specs/{功能名}/tasks.md` 文件中將對應的任務標記完成；本專案目前以 `.agents` 為主

### 5. 任務推薦

- 如果您指定了具體任務，就實作該任務
- 如果沒有指定，我會查看該規格的任務清單並推薦下一個應該執行的任務

### 6. 品質控制

- 完成任務後停下來讓您審查
- **不會自動繼續到列表中的下一個任務**
- 在任務列表中標記完成狀態
- 只有在您要求時才自動執行測試

### 7. 並行操作優化

- 需要執行多個獨立操作時，同時調用所有相關工具而非按順序執行
- 使用 `strReplace` 工具時，將其分解為獨立操作然後同時調用
- 盡可能優先並行調用工具

## 任務問答處理

用戶可能會詢問任務相關問題而不想執行它們。在這種情況下不要總是開始執行任務。

例如，用戶可能想知道特定功能的下一個任務是什麼。在這種情況下，只提供資訊而不開始任何任務。

## 關鍵原則

- **一次一個任務**——完成一個任務後停止，不要自動繼續下一個任務
- **用戶主導**——只有用戶要求時才繼續下一個任務
- **專注執行**——不實作當前任務範圍外的功能
- **持續驗證**——確保實作符合任務要求和設計規範
- **必須展示思考過程**——展示問題分析、方案選擇依據、規範依據、參考程式碼
- **必須複習程式碼分層規範**——程式碼開發任務開始前必須閱讀並展示對應分層的規範
- **必須讀取參考文件**——如果規範指定了參考文件，必須重新讀取這些參考文件

## 完成標誌

所有任務完成並創建 {task}-{task-id}-{few-words-of-the-title}-completion.md 文件記錄完成狀態
