# 第四階段：需求評審

## 階段目标

多維度評估PRD品質，確保可執行性和業務價值，为文件優化提供客观依据。

## 前置條件

- 已完成需求調研、產品規劃和文件撰寫階段
- 存在 `.opencode/prd/{功能名}/draft.md` 文件
- PRD文件草稿已获得使用者初步认可

## 評估維度

### 1. 需求完整性（25%）

評估PRD是否完整覆盖了所有必要的需求和場景。

**評分標準**：

- **功能覆盖度**（10分）
  - 是否覆盖所有核心功能
  - 是否考虑了主要業務場景
  - 是否描述了使用者角色和權限

- **边界條件**（8分）
  - 是否定義了功能边界
  - 是否说明了什么不做
  - 是否考虑了約束條件

- **异常處理**（7分）
  - 是否考虑了异常場景
  - 是否定義了错误處理規則
  - 是否有容错机制

### 2. 可執行性（25%）

評估PRD是否清晰、具體，便于開發团队理解和實作。

**評分標準**：

- **開發理解度**（10分）
  - 需求描述是否清晰无歧义
  - 介面設計说明是否充分
  - 業務規則是否定義明确

- **資源匹配度**（8分）
  - 工作量評估是否合理
  - 是否考虑了技術可行性
  - 是否匹配团队能力

- **驗收標準**（7分）
  - 是否定義了明确的驗收標準
  - 驗收標準是否可測試
  - 是否包含測試方法

### 3. 使用者價值（20%）

評估PRD是否真正解决使用者問題，创造業務價值。

**評分標準**：

- **需求匹配度**（8分）
  - 是否解决了使用者痛点
  - 是否满足業務目标
  - 是否符合使用者期望

- **體驗設計**（7分）
  - 使用者流程是否简洁高效
  - 介面設計是否友好易用
  - 交互反馈是否及时准确

- **價值创造**（5分）
  - 是否能提升業務效率
  - 是否能降低运营成本
  - 是否有明确的ROI

### 4. 業務邏輯（20%）

評估PRD的業務邏輯是否合理、完整、一致。

**評分標準**：

- **流程合理性**（8分）
  - 業務流程是否符合實際
  - 流程設計是否高效
  - 是否考虑了流程優化

- **規則完整性**（7分）
  - 業務規則是否完整
  - 規則定義是否准确
  - 是否考虑了規則冲突

- **資料一致性**（5分）
  - 資料模型是否合理
  - 資料关系是否准确
  - 是否保证資料一致性

### 5. 文件品質（10%）

評估PRD文件本身的品質和規範性。

**評分標準**：

- **結構清晰度**（4分）
  - 文件結構是否合理
  - 章節組織是否清晰
  - 是否易于查找信息

- **描述准确性**（3分）
  - 术语使用是否准确
  - 描述是否准确无误
  - 是否避免了歧义

- **格式規範性**（3分）
  - 格式是否统一規範
  - 圖表是否清晰完整
  - 是否便于阅读理解

## 工作內容

### 1. 全面分析文件一致性

- 檢查 research.md、architecture.md、draft.md 的一致性
- 确认需求調研的問題在PRD中得到解决
- 确认產品規劃的架構在PRD中得到体现
- 识别文件之间的不一致之处

### 2. 多維度評估PRD品質

- 按照評估維度逐項評分
- 記錄发现的問題和改进建議
- 分析問題的優先順序和影響范围
- 提供具體的優化方案

## 品質標準

### 优秀（≥8.5分）

- 需求完整，覆盖全面
- 描述清晰，无歧义
- 業務邏輯合理完整
- 使用者價值明确
- 文件品質高
- 可以直接进入發布准备

### 良好（7.0-8.4分）

- 需求基本完整
- 描述基本清晰
- 業務邏輯基本合理
- 有明确的使用者價值
- 文件品質良好
- 建議優化后發布

### 需要優化（6.0-6.9分）

- 需求不够完整
- 部分描述不清晰
- 業務邏輯有不合理之处
- 使用者價值不够突出
- 文件品質一般
- 建議優化主要問題

### 必須優化（<6.0分）

- 需求缺失较多
- 描述模糊或有歧义
- 業務邏輯存在严重問題
- 使用者價值不明确
- 文件品質较差
- 必須大幅改进

## 完成确认

當評審報告建立完成后，我会：

1. 向您展示評審報告的核心內容
2. 说明综合得分和主要問題
3. 征求您對評審结果的意见
4. 确认是否需要进入優化階段
5. 等待您的明确决定

## 注意事項

- 評審要客观公正，基於事实
- 評分要有依据，不能主观臆断
- 問題要具體明确，不能泛泛而谈
- 建議要可操作，提供具體方案
- 優先順序要准确，反映問題严重程度
- 評審结论要明确，给出清晰建議

## 評審示例

### 評審概要示例

```markdown
## 評審概要

- **評審时间**：2025-10-24
- **評審版本**：draft.md v1.0
- **综合得分**：7.2分（满分10分）
- **評審结论**：良好，建議優化后發布

## 評分详情

### 1. 需求完整性（25%）

| 評估項     | 权重    | 得分    | 说明                                     |
| ---------- | ------- | ------- | ---------------------------------------- |
| 功能覆盖度 | 10%     | 8.0     | 覆盖了核心功能，但缺少供應商自助注册場景 |
| 边界條件   | 8%      | 7.5     | 定義了主要边界，但未明确資料迁移边界     |
| 异常處理   | 7%      | 6.5     | 考虑了基本异常，但网络异常處理不够詳細   |
| **小计**   | **25%** | **7.3** |                                          |

**优点**：

- 核心功能覆盖完整，主要業務場景都已考虑
- 边界定義基本清晰

**問題**：

- 缺少供應商自助注册場景的描述
- 資料迁移边界未明确
- 网络异常處理規則不够詳細

**建議**：

- 补充供應商自助注册場景的需求描述
- 明确資料迁移的范围和方式
- 完善网络异常的處理規則和使用者提示
```

### 主要問題示例

```markdown
## 主要問題

**P0（严重問題，必須修复）**：

1. **供應商評級算法未明确**
   - 影響范围：影響供應商管理核心功能
   - 建議方案：补充評級算法的計算規則和权重分配

**P1（重要問題，建議修复）**：

1. **詢价超时處理規則不清晰**
   - 影響范围：影響詢价协同的使用者體驗
   - 建議方案：明确超时时间和超时后的處理邏輯

2. **价格對比維度不够全面**
   - 影響范围：影響价格對比的决策價值
   - 建議方案：补充品質、交期等維度的综合對比

**P2（次要問題，可选修复）**：

1. **统计报表需求描述过于简單**
   - 影響范围：影響績效管理的可用性
   - 建議方案：詳細描述报表的維度和指标
```
