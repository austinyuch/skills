# 第二階段：六階段流程設計

## 階段概述

基於第一階段的需求分析，設計完整的六階段工作流程，包括每個階段的目标、輸入輸出、評估体系等。

## 觸發條件

- 第一階段的 `requirements.md` 已完成并获得使用者认可
- 或使用者明确表示要进行流程設計

## 階段目标

1. 設計符合需求的六個工作階段框架
2. 为每個階段定義清晰的目标、輸入、輸出
3. 設計專業的多維度評估体系（用于第4階段）
4. 設計流程控制邏輯和階段推进机制
5. 產生完整的流程設計文件

## 具體工作內容

### 2.1 階段框架設計

基於需求分析，設計六個連續的工作階段。使用以下設計表格：

#### 六階段設計模板

| 階段 | 階段名称   | 核心目标           | 輸入條件       | 輸出成果   | 文件名     |
| ---- | ---------- | ------------------ | -------------- | ---------- | ---------- |
| 1    | {分析階段} | 深度理解和需求明确 | 使用者原始需求 | {分析文件} | {file1}.md |
| 2    | {設計階段} | 架構規劃和方案設計 | 分析结果       | {設計方案} | {file2}.md |
| 3    | {创作階段} | 具體內容创作實施   | 設計方案       | {核心成果} | {file3}.md |
| 4    | {評估階段} | 多維度品質評估     | 创作成果       | 評估報告   | review.md  |
| 5    | {優化階段} | 基於評估的改进     | 評估報告       | 優化成果   | {file5}.md |
| 6    | {交付階段} | 最終格式化交付     | 優化成果       | 最終交付   | final.md   |

#### 階段命名原則

1. **第1階段**：通常是"理解"、"分析"、"明确"類型
   - 示例：需求明确、內容构思、問題分析

2. **第2階段**：通常是"設計"、"規劃"、"架構"類型
   - 示例：架構設計、大纲設計、方案規劃

3. **第3階段**：通常是"實施"、"创作"、"實作"類型
   - 示例：內容创作、代码實作、方案執行

4. **第4階段**：固定为"評估"、"复核"、"審查"
   - 固定文件名：review.md

5. **第5階段**：固定为"優化"、"改进"、"迭代"
   - 文件名通常与第3階段相关

6. **第6階段**：通常是"交付"、"發布"、"定稿"
   - 固定文件名：final.md

### 2.2 評估維度設計

为第4階段（品質評估）設計專業的評估体系。應包含4-5個評估維度，每個維度有明确的評分標準。

```markdown

- {評估要点2}
- {評估要点3}

**評分標準（0-10分）**：

- {評估要点3}

[同上結構]

### 综合評分机制

**計算公式**：
```

综合得分 = 維度1得分 × 权重1 + 維度2得分 × 权重2 + ... + 維度N得分 × 权重N

```

**品質等級**：
- **≥8.5分**：优秀 - 可直接进入交付階段
- **6.0-8.4分**：良好 - 建議優化后交付
- **<6.0分**：需改进 - 必須優化

### 優化策略

**第一轮優化**（综合得分 < 8.5）：
- 重点解决評分最低的維度問題
- 针對性改进具體薄弱環節
- 目标：提升至8.0分以上

**第二轮優化**（如果第一轮后仍 < 8.5）：
- 全面打磨細節
- 優化次低分維度
- 目标：达到8.5分

**第三轮優化**（如果仍需要）：
- 最后的精細調整
- 追求完美細節
- 这是最后一轮，之后无论得分如何都将交付
```

#### 不同领域的評估維度示例

**內容创作类**：

1. 內容品質（30%）- 准确性、深度、價值
2. 結構邏輯（25%）- 組織、流畅、连贯
3. 語言表达（25%）- 清晰、專業、易读
4. 使用者體驗（20%）- 实用性、可操作性

**技術開發类**：

1. 需求符合度（25%）- 功能完整性
2. 設計一致性（25%）- 架構符合度
3. 代码規範（25%）- 编码標準
4. 代码品質（25%）- 可维护性、性能

**產品設計类**：

1. 需求准确性（30%）- 理解深度
2. 方案完整性（25%）- 覆盖全面性
3. 可行性（25%）- 實作难度評估
4. 创新性（20%）- 解决方案独特性

### 2.3 流程控制邏輯設計

#### 階段识别机制

```markdown
## 自動階段识别

基於文件存在性判断當前階段：

无文件 → 第1階段（{階段1名称}）
存在{file1}.md → 第2階段（{階段2名称}）
存在{file2}.md → 第3階段（{階段3名称}）
存在{file3}.md → 第4階段（{階段4名称}）
存在review.md → 第5階段（{階段5名称}）或第6階段（根据評分）
存在{file5}.md → 第6階段（{階段6名称}）
存在final.md → 流程完成
```

#### 階段推进确认

每個階段完成时的自然确认方式：

1. **第1階段完成**："{階段1成果}已经完成，{具體內容描述}。准备好进入{階段2名称}了吗？"

2. **第2階段完成**："{階段2成果}設計完成，{具體內容描述}。可以开始{階段3名称}了吗？"

3. **第3階段完成**："{階段3成果}已创作完成。现在让我进行全面的{評估類型}，檢查品質如何？"

4. **第4階段完成**：
   - 高分（≥8.5）："評估得分{X.X}分，品質优秀！可以直接{交付/定稿}了吗？"
   - 中分（6.0-8.4）："評估得分{X.X}分，发现一些改进空间。要優化一下吗？"
   - 低分（<6.0）："評估得分{X.X}分，建議进行優化。让我帮您改进这些問題。"

5. **第5階段完成**："優化完成，得分從{旧分}提升到{新分}。{繼續優化/准备交付}？"

6. **第6階段完成**："{最終成果}已经完成！整個流程圆满结束。"

## 重要約束

1. **六階段完整性**
   - 必須包含完整的六個階段
   - 第4階段必須是評估，第5階段必須是優化
   - 階段之间必須有清晰的輸入輸出关系

2. **評估体系專業性**
   - 評估維度必須与领域特征匹配
   - 評分標準必須明确、可操作
   - 权重分配必須合理（总和100%）

3. **流程邏輯合理性**
   - 階段推进必須基於文件狀態
   - 确认方式必須自然、友好
   - 异常情况必須有處理机制

4. **使用者确认机制**
   - 必須等待使用者的明确认可才能进入下一階段
   - 如果使用者要求修改，必須調整設計并再次确认
   - 必須繼續反馈-修订循环直到获得明确批准

## 完成标志

當满足以下條件时，第二階段完成：

1. ✅ 已建立 `workflow_design.md` 文件
