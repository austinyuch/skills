# 第三階段：文件撰寫

## 階段目标

基於產品規劃建立完整的B端供應鏈PRD文件，詳細描述功能需求、介面設計、業務規則和驗收標準。

## 前置條件

- 已完成需求調研和產品規劃階段
- 存在 `.opencode/prd/{功能名}/research.md` 文件
- 存在 `.opencode/prd/{功能名}/architecture.md` 文件
- 產品規劃方案已获得使用者认可

## 工作內容

### 1. 基於需求調研和產品規劃文件撰寫PRD

- 综合需求調研和產品規劃的內容
- 按照標準PRD模板組織文件結構
- 詳細描述每個功能模組的需求細節
- 补充介面設計和交互说明
- 定義業務規則和驗收標準

### 2. 建立PRD文件草稿

建立 `.opencode/prd/{功能名}/draft.md`，嚴格按照標準PRD模板結構撰寫。

## PRD文件標準結構

请參考 [B端供應鏈PRD模板](../reference/PRD_TEMPLATE.md) 獲取完整的文件結構和撰寫規範。

### 核心章節

1. **版本迭代規劃** - 版本计划和里程碑
2. **業務背景与目标** - 項目背景、使用者場景、業務目标
3. **業務名词** - 關鍵業務术语定義
4. **流程圖** - 業務流程、系統流程、資料流程
5. **功能需求详述与介面設計** - 詳細的功能说明和介面設計
6. **資料模型** - 核心實體定義和关系圖
7. **驗收標準** - 功能驗收的標準和測試方法

## 撰寫要求

### 使用供應鏈專業术语

- 採購管理：採購申请、採購订單、供應商評估、合同管理
- 庫存管理：庫存周转、安全庫存、ABC分类、庫存预警
- 物流管理：運輸管理、配送路線、運費結算、貨物跟踪
- 供應商协同：订單协同、发货通知、對账結算、績效評估

### 每個功能明确業務價值和ROI

为每個功能模組说明：

- **業務價值**：解决什么問題，带来什么收益
- **投入产出比**：预估的成本和收益對比
- **優先順序理由**：为什么要优先做这個功能

### 预留圖片位置和插入提示

在需要插入圖片的地方使用占位符：

```markdown
圖片位置：`./images/供應商管理頁面.png`

> 📋 **提示**：粘贴供應商管理頁面原型圖
```

### 关注敏捷迭代和現有系統集成

- **功能原子化**：每個功能点可独立開發、測試、上線
- **最小侵入**：减少對現有系統的改動
- **向下兼容**：確保新功能不影響現有業務
- **資料一致**：新旧系統資料保持一致

**使用者故事**：作为[角色]，我希望[功能]，以便于[價值]

**頁面布局**：

**業務規則**：

- **資料驗證規則**：供應商编码唯一性校验、联系方式格式校验

- **狀態流轉規則**：新建→審核→启用→禁用的狀態流轉
- **异常處理規則**：网络异常时顯示离線缓存資料

**驗收標準**：

| 驗收項目   | 驗收標準                   | 測試方法            |
| ---------- | -------------------------- | ------------------- |
| 功能完整性 | 所有使用者故事場景正常執行 | 手工測試+自動化測試 |
| 性能要求   | 頁面加载时间<2秒           | 性能測試工具驗證    |

## 注意事項

- PRD撰寫要基於需求調研和產品規劃
- 要使用供應鏈行業專業术语

## 常见問題

### Q: PRD應该写到什么程度的細節？

A: PRD應该詳細到：

- 開發人员能够理解要做什么功能
- 設計师能够設計介面原型
- 測試人员能够编写測試用例
- 產品运营能够理解業務邏輯

但不應该包括：

- 具體的技術實作方案
- 資料庫表結構設計
- API接口定義
- 代码實作細節

### Q: 如何平衡文件詳細度和敏捷開發？

A: 建議采用：

- **核心功能詳細描述**：P0和P1功能要詳細描述
- **扩展功能简要说明**：P2和P3功能可以简要说明
- **支援渐进式完善**：随着開發推进，补充完善細節
- **保持文件最新**：及时更新PRD反映最新决策

### Q: 如何處理需求變更？

A: 建議：

- 在PRD中記錄變更歷史
- 说明變更原因和影響范围
- 更新相关的功能描述
- 重新評審和确认
