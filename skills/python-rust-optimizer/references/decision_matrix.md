# 決策矩陣：何時使用 Rust 優化 Python

## 資料大小閾值

| 資料大小 | 建議 | 原因 |
|---------|------|------|
| < 1KB | 保留 Python | Python/Rust 邊界開銷 > 效能增益 |
| 1KB - 100KB | 視情況 | 需要 benchmark 驗證 |
| > 100KB | 使用 Rust | 計算密集，Rust 優勢明顯 |

## 運算類型評估

### 高優先級 (強烈建議 Rust)

- ✅ NumPy 密集運算 (大陣列操作)
- ✅ 數值計算 (線性代數、統計)
- ✅ 影像處理 (OpenCV 替代)
- ✅ 加密/解密
- ✅ 壓縮/解壓縮
- ✅ 解析器 (JSON/XML/Binary)
- ✅ 正則表達式 (大量文字)

### 中優先級 (需要 benchmark)

- ⚠️ List comprehension (大資料集)
- ⚠️ 字串處理 (大量操作)
- ⚠️ 資料轉換 (DataFrame 操作)
- ⚠️ 排序/搜尋 (大資料集)

### 低優先級 (保留 Python)

- ❌ I/O 密集操作
- ❌ 網路請求
- ❌ 資料庫查詢
- ❌ 簡單的業務邏輯
- ❌ 小型資料結構操作

## GIL 影響評估

### 高 GIL 影響 (建議 Rust + allow_threads)

```python
# Pattern: CPU 密集型迴圈
for i in range(1000000):
    result = complex_calculation(i)  # 阻塞 GIL
```

**解決方案**: Rust + `py.allow_threads()` 釋放 GIL

### 中 GIL 影響 (視情況)

```python
# Pattern: NumPy 操作 (已釋放 GIL)
result = np.dot(large_matrix_a, large_matrix_b)
```

**評估**: NumPy 已優化，但 Rust 可能更快

### 低 GIL 影響 (保留 Python)

```python
# Pattern: I/O 操作
with open('file.txt') as f:
    data = f.read()  # I/O 自動釋放 GIL
```

## 效能分析閾值

根據 `cProfile` 分析結果：

| 累積時間佔比 | 建議 | 行動 |
|-------------|------|------|
| > 30% | 強烈建議 | 立即轉換為 Rust |
| 20-30% | 建議 | Benchmark 後決定 |
| 10-20% | 可選 | 低優先級優化 |
| < 10% | 不建議 | 保留 Python |

## 效能增益預期

### 預期加速比

| 場景 | 預期加速 | 實際案例 |
|------|---------|---------|
| NumPy 密集運算 | 2-5x | 矩陣運算: 3.8x |
| 純 Python 迴圈 | 10-100x | 數值計算: 45x |
| 字串處理 | 3-10x | 正則表達式: 7.2x |
| 並行處理 | 4-8x | Rayon 並行: 6.1x |

### 最小效能增益要求

- **開發環境**: 至少 2x 加速才值得轉換
- **生產環境**: 至少 3x 加速才值得維護成本
- **關鍵路徑**: 至少 5x 加速 (考慮複雜度)

## 決策流程圖

```
開始
  ↓
是否為 CPU 密集型? ──否→ 保留 Python
  ↓ 是
累積時間 > 20%? ──否→ 低優先級
  ↓ 是
資料大小 > 100KB? ──否→ 執行 benchmark
  ↓ 是
是否有 NumPy 操作? ──是→ 高優先級 (Rust + NumPy)
  ↓ 否
是否為純 Python 迴圈? ──是→ 極高優先級 (轉換 Rust)
  ↓ 否
是否需要並行? ──是→ 高優先級 (Rust + Rayon)
  ↓ 否
中優先級 (benchmark 後決定)
```

## 實際案例參考

### 案例 1: NumPy 矩陣運算

- **場景**: 大型矩陣乘法
- **資料大小**: 1000x1000 矩陣
- **累積時間**: 45%
- **決策**: ✅ 轉換為 Rust
- **結果**: 3.8x 加速

### 案例 2: JSON 解析

- **場景**: API 回應解析
- **資料大小**: 平均 50KB
- **累積時間**: 8%
- **決策**: ❌ 保留 Python
- **原因**: 邊界開銷 > 效能增益

### 案例 3: 影像濾鏡

- **場景**: 即時影像處理
- **資料大小**: 1920x1080 像素
- **累積時間**: 62%
- **決策**: ✅ 轉換為 Rust (SIMD)
- **結果**: 8.5x 加速

### 案例 4: 資料清洗

- **場景**: Pandas DataFrame 操作
- **資料大小**: 100萬行
- **累積時間**: 25%
- **決策**: ⚠️ Benchmark 後決定
- **結果**: 2.1x 加速 (邊緣案例)

## Python 特定考量

### 何時保留 Python

- **動態型別需求**: 需要 duck typing
- **快速原型**: 開發速度優先
- **生態系統**: 需要 Python 特定庫
- **可讀性**: 團隊不熟悉 Rust
- **維護成本**: 小型專案不值得

### 何時使用 Rust

- **效能關鍵**: 生產環境瓶頸
- **並行需求**: 需要真正的並行
- **記憶體效率**: 大資料處理
- **型別安全**: 需要編譯時檢查
- **長期維護**: 大型專案值得投資
