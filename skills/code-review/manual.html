Ponytail / YAGNI Boundary

Where it applies

Use the ladder before recommending new abstractions, helpers, dependencies, wrappers, parsers, lifecycle managers, class/module splits, or test harnesses. It supports design and implementation decisions, not Phase 5 verdicts.

How to report it

Report it through code-refactoring-advisor as advisory minimality_check context: smell, refactor move, safety net, and the smallest portable alternative. Do not translate it into PASS, FAIL, BLOCK, or approval language.

Common Walkthroughs

A

Analyze a file

For Go/Python/TypeScript/C#, run the built-in AST review (review-cli review <path>) as the primary deterministic pass — it needs no LLM and is accurate by construction. Then summarize the main structure, risks, and next handoff.

B

Review tests

Hand the test file to `test-quality-reviewer` and keep the output scoped to test quality.

C

Run security preflight

Use `security-risk-reviewer` for OWASP-oriented checks and keep the result ranked, not final.

CLI / Artifact Interpretation

Good outputs

Deterministic findings, ranked issues, structured JSON, and short handoff notes that name the next action clearly.

What to avoid

Do not turn graph output or helper output into a production verdict. Use it as input to the next review step.

Claim boundary: Code Review provides review evidence and bounded-context handoffs. Readiness still belongs to the consuming repo's review artifacts and runtime evidence.

FAQ

When should I use code-review?

When the task is about code structure, tests, refactoring, security, or Sonar evidence and you need a review-oriented handoff.

Is this a runtime gate?

No. It is static analysis plus bounded-context tooling.

Why are the helper skills separate?

Because tests, refactors, security, and capability summaries are distinct review problems and should not be collapsed into one opaque workflow.

Responsible AI Operator Checks

Before accepting findings

Confirm each finding has an explainable risk, evidence source, affected file or behavior, and next owner. Treat tool output as review input, not final approval.

Before handoff

Check for secrets, privacy-sensitive data, auth or tenant-boundary risk, and stale scanner output before handing the report to another agent or team.

Evidence And Claim Boundary

This manual explains how to run the review family and interpret its outputs. It does not claim that a consuming repository is production-ready, demo-ready, or fully integrated.
