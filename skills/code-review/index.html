The Code Review Family

Make review deterministic about evidence — not just verbose about opinions

A single "review this" prompt gives opinions that vary run to run and bury the dangerous finding under style nits. This family turns "review it properly" into deterministic, risk-ranked, actionable findings — the verify (left-shift) layer of The Spec Master Method.

Why a family, not one reviewer

Two principles make reviews trustworthy evidence instead of variable opinion.

Deterministic-first, LLM-optional

Each helper ships a deterministic detector that is the source of truth, needs no provider, and is safe in CI. An optional --explain / LLM pass only calibrates severity and writes reasoning — it never invents findings.

Risk-based triage

Findings are ranked (e.g. severity × confidence × blast-radius) so review effort goes to the highest-impact issue first. What to review is judgement, not generation.

Family members

Skill Question it answers What it owns
code-reviewWhat does this code look like — structure, dependencies, impact, bounded context?Orchestrator + program-graph / GraphRAG static analysis; folds in every sibling's findings. CLI binary with an MCP runtime surface.
code-refactoring-advisorWhat should I change, and what test net must exist first?Deterministic Fowler smell detection → a named refactoring move + the test safety-net required before the move is safe.
test-quality-reviewerAre these tests any good — would a real defect survive them?Judges existing tests: assertion roulette, mystery guest, fragile/conditional tests, FIRST violations, pyramid inversion. Green only means it ran.
test-design-generatorWhat should I test — what are the real edge cases?Generates boundary-value, equivalence-partition, and pairwise cases from a parameter description; recommends an oracle strategy when outputs can't be enumerated.
security-risk-reviewerIs this secure — where's the injection / secret / weak crypto?Deterministic OWASP Top-10 scan, ranked by risk score so the dangerous lines surface first. Each finding mapped to OWASP/CWE.
sonarqube-bridgePull in our SonarQube findings and rank them with everything else.An adapter, not a scanner: ingests Sonar findings and enriches each with code-graph blast-radius, business capability, and remediation vocabulary. Never scans, never gates.

How they compose

Inventory → improvement

code-review builds the inventory ("what exists") — the program graph, traceability, bounded context. The helpers extend it to "what should improve".

Change ↔ safety net

code-refactoring-advisor says what to change; test-quality-reviewer checks the net you change it against; test-design-generator produces the missing cases when a test is thin.

Executable security + Sonar

security-risk-reviewer makes prose security guidance executable. sonarqube-bridge is connective tissue: Sonar owns the blocking Quality Gate; this family adds Sonar's blind spots and fuses both into one ranked report.

Scope boundary

This family is static analysis + review evidence, NOT a runtime readiness authority. Use it for structure, impact, what-should-improve, test effectiveness, and ranked security risk. Do NOT use it alone to claim a feature is live-demo ready — authoritative PASS / CONDITIONAL / FAIL still comes from runtime-backed review.md. Static green ≠ shipped.

Place in the method

In the six-phase flow this family is Phase 4/5 left-shift detection. spec-driven-development runs the lifecycle and pulls these skills in as capability-gated inputs at implementation and review. review.md still owns the verdict — this family makes that verdict rest on deterministic evidence instead of one-shot opinion.
