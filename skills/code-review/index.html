Code Review Family

One review surface for code, tests, and security handoff

The family gives coding agents a governed way to inspect structure, rank impact, surface test and security gaps, and hand off bounded context without treating static analysis as a release gate.

First Principles

Behavior shift

Agentic AI moves review pressure from a single human reading code to a mixed human-agent loop where defects can move faster than context.

Role lens

CEO value is faster delivery with less duplicate effort; CIO value is a reusable review contract; DevSecOps value is stronger evidence integrity. The matching risks are hidden quality debt, authority drift, and false green output.

Solution mechanics

Review must become more deterministic about evidence, not just more verbose about opinions. Code Review does that by separating diagnosis, refactor advice, test design, and security preflight, while keeping graph and summary tooling as inputs rather than verdicts.

Team-owned workflow

Coding agents change quickly, vendors make different orchestration choices, and many review decisions remain partly black-box. Code Review gives the team an inspectable review contract, evidence trail, and retrospective surface so quality improvement belongs to the engineering team.

Ponytail / YAGNI Minimality Boundary

When To Use

Use it when

You need a review front door for a file, directory, or project and want one contract for structure, impact, and handoff.

The repo needs to absorb test, refactor, security, and scanner evidence into one review path.

Do not use it when

The task is only to write code without review analysis.

The main work is already a narrow test, refactor, or security task owned by a sibling helper.

What You Lose If You Merge These

Test diagnosis + design

If `test-quality-reviewer` and `test-design-generator` are merged, the team gets either vague suggestions or test ideas that miss the real coverage gap.

Review + refactor advice

If `code-refactoring-advisor` is merged into `code-review`, refactor prescriptions start arriving before the underlying defect analysis is clear.

Security + main review

If `security-risk-reviewer` is merged with the orchestrator, security triage loses its own vocabulary and the next owner gets a less reusable handoff.

Review Depth

Deterministic lane

Structure and impact

The built-in AST review and graph-retrieval surfaces answer what exists, how it connects, and where the blast radius is likely to land.

Sibling helpers

Tests and refactors

`test-quality-reviewer`, `code-refactoring-advisor`, and `test-design-generator` keep diagnosis, design, and remediation separate.

Security + SAST bridge

Risk triage

`security-risk-reviewer` and `sonarqube-bridge` fold security evidence into the same ranked handoff without pretending to replace the authoritative scanner.

Outcome Signals

Fewer repeated comments

The family is working when the same code path is not explained repeatedly in different review threads.

Cleaner evidence chains

Test, refactor, security, and scanner signals stay separate enough to be reused by the next owner.

Less false confidence

Readiness language stays outside the review bundle, so evidence is less likely to be overstated.

FAQ

Who is this for?

Teams that need review help on real repositories, especially when code structure, tests, and security need to be analyzed together.

Is this a runtime readiness gate?

No. It is static analysis and bounded-context tooling. Runtime readiness still comes from the consuming repo's review and runtime evidence.

Why separate test-quality and test-design?

One judges existing tests; the other generates candidate tests. Keeping them separate preserves the distinction between diagnosis and design. This matters because teams often confuse "we need more tests" with "our current tests are weak in these specific ways." The two actions have different costs and different owners.

Why include `sonarqube-bridge`?

Because Sonar issues are often the same review conversation in a different source system. Bridging them into the same ranking model keeps the handoff coherent.

Are `capability-mapper` and `code-summarizer` core family members?

No. They are supporting tooling that the family can orchestrate when graph or summary context is useful.

What should be packaged with this family?

At minimum, package `skills/code-review/`, its sibling review skills, and the supporting tooling if the target environment expects graph or summary enrichment.

Evidence And Claim Boundary

This README describes the review capability bundle. It does not claim that a consuming repository is production-ready, demo-ready, or free of regressions. Those claims belong to the consuming repo's own review artifacts and runtime evidence.
