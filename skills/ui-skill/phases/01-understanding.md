# 階段一：需求理解階段詳細指南

## 階段目標

深度理解使用者需求，明确要產生的介面功能和范围。

## 輸入條件

使用者提供以下任一輸入：

- 功能描述（簡單描述或詳細說明）
- PRD markdown檔案

## 頁面產生原則

1. 如果需要產生一組多個頁面
   - 必須有一個index.html檔案，作為主頁面串联所有頁面。
   - 每個頁面必須有獨立的HTML檔案。
   - 所有頁面必須使用相同的UI模板風格。
   - 注意多個頁面之间的跳轉。
   - 多個頁面必須邏輯闭环，支援点击頁面内的按钮或連結跳轉到其他頁面。

## 執行步骤

### 步骤1：识別輸入類型

**判断標準**：

- 如果使用者提供了markdown檔案或包含明显檔案結構的內容 → PRD檔案
- 如果使用者簡單描述功能（如"供應商清單格頁"） → 功能描述

**處理方式**：

- **PRD檔案**：先通读全文，重点关注【功能需求详述與介面設計】章節
- **功能描述**：直接進入需求提取

### 步骤2：需求提取和分析

#### 2.1 提取核心功能模組

识別頁面的核心功能模組，常见的B端供應鏈功能模組包括：

**資料展示類別**：

- 資料清單格/表格格
- 统計卡片/仪表格板
- 資料詳情

**資料操作類別**：

- 新增/建立
- 編輯/修改
- 刪除
- 批量操作

**資料篩選類別**：

- 搜尋
- 篩選條件
- 排序
- 分頁

**業務流程類別**：

- 审批流程
- 狀態流轉
- 匯入匯出

#### 2.2 识別頁面類型

確定需要產生哪些類型的頁面：

- **清單格頁**：展示資料清單格，包含篩選、搜尋、分頁等功能
- **詳情頁**：展示單条資料的詳細信息
- **表格單頁**：新增或編輯資料的表格單頁面
- **仪表格板頁**：统計資料和圖表格展示

#### 2.3 识別資料欄位和結構

提取關鍵的資料欄位：

- 欄位名称
- 欄位類型（文本、數字、日期、枚举等）
- 是否必填
- 欄位說明

示例：

```javascript
{
  order_no: "订單編號（文本，必填）",
  supplier: "供應商（枚举，必填）",
  total_amount: "订單金額（數字）",
  status: "狀態（枚举：draft/pending/approved等）",
  apply_date: "申请日期（日期）"
}
```

#### 2.4 识別互動要求

明确需要的互動功能：

**基础互動**：

- 頁面載入和資料渲染
- 回應式佈局

**篩選互動**：

- 搜尋框
- 下拉篩選
- 日期篩選
- 清除篩選

**CRUD互動**：

- 新增資料（打开模態框，填写表格單，提交）
- 編輯資料（打开模態框，填充資料，修改，提交）
- 檢視詳情（打开模態框或跳轉詳情頁）
- 刪除資料（確認對话框，刪除）
- 批量操作（批量選擇，批量刪除/审批等）

**表格單互動**：

- 表格單驗證（必填項、格式校验）
- 表格單提交
- 錯誤提示

**狀態管理**：

- 狀態切换
- 狀態標籤顯示

**其他互動**：

- 匯入匯出
- 打印
- 訊息提示
- 載入動画

### 步骤3：需求確認（互動式梳理）

**重要**：這是本階段的核心步骤，必須與使用者充分互動。

#### 3.1 逐条確認需求理解

不要一次性拋出所有問題，而是分組、有序地確認：

**第一轮確認：頁面范围**

```
"根据我的理解，這個功能需要產生以下頁面：
1. [頁面1名称] - [簡要說明]
2. [頁面2名称] - [簡要說明]
3. ...

這樣理解對吗？还是只需要產生其中的某些頁面？"
```

**第二轮確認：核心功能**

```
"對於[頁面名称]，我理解的核心功能包括：
- [功能1]
- [功能2]
- [功能3]

這些功能完整吗？有遗漏或不需要的吗？"
```

**第三轮確認：資料欄位**

```
"我提取到以下關鍵資料欄位：
- [欄位1]：[類型]、[說明]
- [欄位2]：[類型]、[說明]
- ...

這些欄位准确吗？有需要补充或修改的吗？"
```

**第四轮確認：特殊要求**

```
"关於互動和顯示，有以下特殊要求吗？
- 是否需要批量操作？
- 是否需要匯入匯出功能？
- 是否需要审批流程？
- 資料是否有狀態流轉？
- 其他特殊要求？"
```

#### 3.2 詢问不明确的地方

如果PRD或功能描述中有模糊的地方，主動詢问：

- "PRD中提到'订單管理'，具體是指哪些功能呢？"
- "狀態流轉的具體規則是什么？"
- "篩選條件有哪些？"
- "表格單驗證規則是什么？"

#### 3.3 確認頁面拆分方案

```
"关於頁面拆分，我建議：
- [方案1]：單頁面實作所有功能（清單格+表格單都在一個HTML中）
- [方案2]：多頁面實作（清單格頁、詳情頁、表格單頁分开）

您更倾向於哪種方式？或者有其他建議？"
```

## 輸出成果

- 建立 `.opencode/ui-generator/{功能名}/input.md` 檔案
- 功能名使用kebab-case格式（如"purchase-order"、"supplier-management"）

## 完成標準

必須满足以下所有條件才能進入下一階段：

- ✅ 明确瞭解要產生几個HTML檔案
- ✅ 清楚每個頁面的核心功能
- ✅ 明确了資料欄位和結構
- ✅ 使用者確認需求理解准确
- ✅ 已建立需求理解檔案（input.md）

## 評估維度

- **需求理解准确性（40%）**：是否准确理解使用者意圖
- **需求完整性（40%）**：是否遗漏關鍵功能
- **沟通有效性（20%）**：互動式梳理是否有效解決了疑问

## 常见問題處理

### Q1：使用者提供的描述非常簡單怎么办？

主動提问，幫助使用者細化需求：

```
"您提到需要'供應商清單格頁'，讓我幫您梳理一下具體需求：
1. 這個頁面需要展示哪些供應商信息？（名称、联系方式、合作狀態等）
2. 需要哪些篩選條件？（按名称搜尋、按狀態篩選等）
3. 需要哪些操作？（新增、編輯、刪除、檢視詳情等）
4. 是否需要批量操作？
5. 是否需要匯入匯出功能？"
```

### Q2：PRD檔案太长，如何高效處理？

1. 先速读全文，把握整體結構
2. 重点关注【功能需求详述與介面設計】章節
3. 提取關鍵信息（頁面、功能、欄位、互動）
4. 與使用者確認是否需要實作所有PRD中的功能

### Q3：使用者需求前後矛盾怎么办？

主動指出矛盾，幫助使用者澄清：

```
"我注意到在需求描述中，第3点提到'只顯示审批透過的订單'，
但在篩選條件中又包含'所有狀態'的篩選选項。
请问是：
A) 預設只顯示审批透過的，但可以透過篩選檢視其他狀態
B) 應該去掉狀態篩選，始終只顯示审批透過的
您的意圖是哪種呢？"
```

### Q4：使用者说"參考某個現有系統"怎么办？

如果無法存取該系統，请使用者提供：

- 系統截圖
- 功能描述檔案
- 或者描述具體想要參考的功能点

## 注意事項

1. **不要假設**：不確定的地方一定要问清楚，不要根据经验假設
2. **不要遗漏**：即使是看似簡單的功能，也要確認完整的功能清單
3. **不要跳过確認**：必須與使用者互動式確認，不能自己理解了就直接進入下一階段
4. **保持專業**：即使使用者的需求有不合理之处，也要專業地指出並提供建議
5. **記錄詳細**：所有確認的內容都要詳細記錄在input.md中

## 示例場景

### 示例1：處理簡單功能描述

**使用者輸入**："產生一個採購订單清單格頁面"

**處理過程**：

```
我：好的！讓我幫您產生採購订單清單格頁面。首先讓我瞭解一下具體需求：

1. 关於展示的订單信息，您需要顯示哪些欄位呢？
   - 订單編號
   - 供應商
   - 申请部门
   - 订單金額
   - 订單狀態
   - 申请日期
   - 其他欄位？

2. 关於篩選功能，需要哪些篩選條件？
   - 按订單編號搜尋
   - 按供應商篩選
   - 按狀態篩選
   - 按日期范围篩選
   - 其他？

3. 关於操作功能，需要哪些操作？
   - 新增订單
   - 編輯订單
   - 檢視订單詳情
   - 刪除订單
   - 批量操作（批量审批、批量刪除等）
   - 匯入匯出
   - 其他？

4. 這個頁面是單獨一個HTML檔案（所有功能都在一個頁面），
   还是需要拆分成多個HTML檔案（清單格頁、詳情頁、新增/編輯頁）？

请您詳細說明一下這些需求，我会据此產生完整的原型。
```

**使用者輸入**：[附上PRD檔案]

## 推進到下一階段

完成本階段後，使用以下方式確認推進：

```
"需求理解檔案已產生，我已经明确了：
- 需要產生 [N] 個HTML頁面
- 每個頁面的核心功能
- 所有資料欄位和結構
- 特殊的互動要求

需求理解准确吗？如果確認無误，我們可以進入介面規劃階段，
開始設計頁面佈局和互動流程。"
```
