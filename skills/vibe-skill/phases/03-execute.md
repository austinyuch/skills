# 第三階段：執行 (Execute)

## 觸發條件

- 規劃階段完成
- 使用者確認實作方案和任務清單格
- 准备開始編碼實作

## 階段目標

按照規劃的任務清單格，高效、高品質地完成所有開發任務。

## 我的工作

### 1. 任務執行流程

對於每個任務，遵循以下流程：

1. 顯示任務編號和描述
2. 詳細說明要做的工作內容和具體步骤
3. 開始實際的開發工作
4. 完成後更新任務狀態
5. 簡要說明完成情况
6. 繼續下一個任務

### 2. 任務開始前

每個任務開始前，必須說明：

- 任務編號和名称
- **思考過程** - 為什么要這樣做，决策依据是什么
- 工作內容
- 具體步骤

**重要：必須展示思考過程和决策依据**

在開始任何任務之前，需要清晰展示：

1. **問題分析** - 這個任務要解決什么問題？
2. **方案選擇依据** - 為什么選擇這種實作方式？有哪些备选方案？
3. **規範依据** - 參考了哪些規範規則？規範中的具體条款是什么？
4. **參考程式碼** - 參考了哪些現有程式碼？為什么參考這些程式碼？

示例：

```
任務 2.3 Service 層開發

【思考過程】
- 問題分析：需要實作 xxx 業務邏輯，涉及到 yyy 的處理
- 方案選擇依据：參考現有的 ZzzService 實作模式，因為業務場景類別似
- 規範依据：根据 `.kiro/steering/service-layer.mdc` 規範，Service 層應該...
- 參考程式碼：參考 `app/Services/ZzzService.php` 的實作方式

工作內容：建立 XxxService，實作核心業務邏輯

具體步骤：
1. 在 app/Services 目錄下建立 XxxService.php
2. 注入 XxxRepository 依赖
3. 實作 xxx 方法
4. 實作 yyy 方法
```

### 3. 程式碼分層規範复习（重要）

**在每個程式碼開發任務開始前，必須執行以下規範复习流程：**

#### 3.1 复习流程

對於涉及程式碼編写的任務，必須：

1. **识別程式碼分層** - 明确當前任務涉及哪些程式碼分層（Controller、Service、Repository、Model、Enum 等）
2. **讀取對應規範** - 讀取 `.kiro/steering` 目錄下對應分層的規範檔案
3. **讀取參考檔案** - 如果規範檔案中指定了參考檔案，**必須重新讀取這些參考檔案**
4. **展示規範要点** - 向使用者展示該分層的核心規範要点
5. **確認遵循** - 在開發過程中嚴格遵循這些規範

#### 3.2 規範复习輸出格式

在開始程式碼開發任務前，必須輸出以下內容：

```markdown
【規範复习】

涉及程式碼分層：Service 層

已讀取規範檔案：

- `.kiro/steering/service-layer.mdc`

規範指定的參考檔案（已重新讀取）：

- `app/Services/ExampleService.php`
- `app/Services/AnotherService.php`

核心規範要点：

1. Service 層負責業務邏輯處理
2. 必須注入 Repository 依赖
3. 方法命名規範：xxx
4. ...

【開始開發】
```

#### 3.3 多分層任務處理

如果一個任務涉及多個程式碼分層，必須：

- 逐個复习每個分層的規範
- 展示每個分層的核心要点
- 按業務模組的鏈路順序開發：路由 → 控制器 → 服務層 → 仓储層 → Model層

#### 3.4 規範衝突處理

如果发現規範之间存在衝突或不明确：

- 暂停開發
- 向使用者說明衝突点
- 获得使用者確認後繼續

### 4. 程式碼實作規範

#### 程式碼規範

- 嚴格遵守 `.kiro/steering` 中定義的規範
- 参照規則檔案中指定的參考檔案
- 保持與現有程式碼風格一致

#### 開發順序

按業務模組的核心鏈路開發：路由 → 控制器 → 服務層 → 仓储層 → Model層

按需补充其他分層：枚举、請求驗證、資源转换等

#### 編碼要求

- 严禁硬編碼，使用枚举替代
- 適配多語言，枚举層維護翻译
- 業務異常碼使用數字
- 添加必要的註解說明

### 5. 任務完成後

每個任務完成後，簡要說明：

- 完成了什么
- 檔案位置
- 然後繼續下一個任務

### 6. 進度更新

實時更新 To-dos 狀態：

- 使用 TodoWrite 工具更新任務狀態
- 當前執行的任務標記為 in_progress
- 完成的任務標記為 completed
- 保持使用者對進度的可见性

### 7. 並行執行最佳化

當多個任務相互獨立時：

- 可以同時執行多個工具调用
- 提高執行效率
- 但要確保不会產生衝突

### 8. 簡要自查

所有任務完成後，進行簡要自查：

- 功能完整性檢查
- 規範一致性檢查
- 列出建立/修改的檔案清單

## 重要約束

- **必須顯示任務編號** - 每個任務開始前顯示編號
- **必須展示思考過程** - 展示問題分析、方案選擇依据、規範依据、參考程式碼
- **必須复习程式碼分層規範** - 程式碼開發任務開始前必須讀取並展示對應分層的規範
- **必須讀取參考檔案** - 如果規範指定了參考檔案，必須重新讀取這些參考檔案
- **必須說明具體步骤** - 不能直接写程式碼不解釋
- **必須實時更新進度** - 讓使用者瞭解當前狀態
- **必須遵守規範** - 嚴格按照項目規範編碼
- **不自動執行SQL** - SQL 脚本由使用者手動執行

## 特殊情况處理

### 遇到問題

如果執行過程中遇到問題：

- 暂停執行，向使用者說明情况
- 分析問題原因
- 提出解決方案
- 获得使用者確認後繼續

### 需求變更

如果使用者提出需求變更：

- 暂停當前任務
- 回到規劃階段評估影響
- 更新任務清單格
- 获得確認後繼續執行

### 任務过大

如果发現某個任務过大：

- 將其拆分為多個子任務
- 更新任務清單格
- 先產生框架再补充細節

## 與使用者的互動

執行階段的互動示例：

**任務進行中**：

- "任務 2.3 已完成，繼續執行 2.4..."
- "這個任務比较複雜，我分两步来做..."

**遇到問題**：

- "在實作 xxx 時发現一個問題，需要和你確認一下..."
- "這里有两種實作方式，你倾向於哪種？"

**全部完成**：

- "所有任務都完成了，我做了一個簡單的自查，你可以驗證一下功能..."

## 完成標志

- 所有 To-dos 任務完成
- 簡要自查透過
- 向使用者汇报完成情况
- 使用者確認功能可用
