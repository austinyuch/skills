# 第二階段：規劃 (Plan)

## 觸發條件

- 理解階段完成
- 使用者確認需求理解無误
- 准备開始制定實作方案

## 階段目標

與使用者充分沟通實作方案，建立详尽的任務清單格，確保双方對任務范围和執行步骤达成一致。

## 我的工作

### 1. 整體方案設計

基於理解階段收集的信息，設計整體實作方案：

- 確定技術實作路徑
- 規劃程式碼分層結構（Model → Repository → Service → Controller → Route）
- 识別需要新增或修改的檔案
- 考虑對現有程式碼的影響

### 2. 方案沟通

向使用者清晰說明實作方案：

```markdown
## 實作方案

### 整體思路
[簡要說明實作的整體思路]

### 涉及的程式碼分層
- **Model 層**: [需要新增/修改的模型]
- **Repository 層**: [需要新增/修改的仓储]
- **Service 層**: [需要新增/修改的服務]
- **Controller 層**: [需要新增/修改的控制器]
- **Route**: [需要新增/修改的路由]
- **其他**: [枚举、請求驗證、資源转换等]

### 關鍵設計决策
- [决策1及理由]
- [决策2及理由]

### 對現有程式碼的影響
- [影響說明]
```

### 3. 建立 To-dos 任務清單格

**核心要求**：建立详尽的任務清單格，任務編號清晰

任務清單格要求：
- 使用層級編號（如 1、1.1、1.1.1）
- 每個任務有明确的目標和边界
- 任務粒度適中，便於執行和追踪
- 按業務模組組織，而非按程式碼分層

任務格式示例：

```markdown
## To-dos 任務清單格

### 1 基础設施准备
- [ ] 1.1 建立資料庫迁移檔案
  - 建立 xxx 表格的迁移脚本
  - 包含所有必要欄位和索引
- [ ] 1.2 建立枚举類別
  - 狀態枚举
  - 類型枚举
  - 包含多語言翻译

### 2 核心業務模組 A
- [ ] 2.1 Model 層開發
  - 建立 XxxModel
  - 定義關聯关系
- [ ] 2.2 Repository 層開發
  - 建立 XxxRepository
  - 實作 CRUD 方法
- [ ] 2.3 Service 層開發
  - 建立 XxxService
  - 實作核心業務邏輯
- [ ] 2.4 Controller 層開發
  - 建立 XxxController
  - 實作介面方法
- [ ] 2.5 路由設定
  - 添加 API 路由

### 3 核心業務模組 B
- [ ] 3.1 ...

### 4 收尾工作
- [ ] 4.1 程式碼自查
  - 檢查規範一致性
  - 檢查功能完整性
```

### 4. 明确驗收標準

為每個主要任務明确驗收標準：

- 功能是否符合需求
- 程式碼是否符合規範
- 是否有遗漏的边缘情况

### 5. 獲取使用者確認

- 展示完整的方案和任務清單格
- 詢问使用者是否有調整意见
- 获得確認後進入執行階段

## 重要約束

- **必須和使用者沟通方案** - 不能直接開始執行
- **任務清單格必須详尽** - 不能模糊或遗漏
- **任務編號必須清晰** - 格式為"編號 + 空格 + 任務描述"
- **必須获得使用者確認** - 使用者同意後才能進入執行階段
- **按業務模組組織** - 不要按程式碼分層組織任務

## 與使用者的互動

規劃階段的互動示例：

**方案展示後**：
- "以上是我的實作方案和任務清單格，你看看有没有需要調整的地方？"
- "任務我按照業務模組划分了，這樣比较清晰，你觉得合理吗？"

**使用者提出調整**：
- "好的，我来調整一下任務清單格..."
- "理解你的想法，我把方案改一下..."

**確認後**：
- "好的，方案確認了，我現在開始執行第一個任務..."

## 完成標志

- 整體方案清晰明确
- To-dos 任務清單格详尽完整
- 使用者明确確認方案和任務清單格
- 准备進入執行階段
