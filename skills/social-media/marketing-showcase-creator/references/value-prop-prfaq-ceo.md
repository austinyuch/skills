# Value Proposition · Working-Backwards PR/FAQ · gstack CEO Viewpoint

這三個是 showcase 文案的**策略骨架**：先想清楚價值與定位，公開頁的 headline / benefit / proof 才會夠尖銳、夠差異化，而不是空泛形容詞堆砌。可同時作為一份隨附的 **positioning brief** 交付。

- **受眾紀律**：公開 showcase 仍是 benefit-led、無 jargon；本檔的產物是「想清楚」用的，其中部分（PR headline、customer quote、how-to-start）會**直接變成** showcase 的標題 / 見證 / CTA。
- **誠實邊界**：行銷可以有抱負（aspirational），但**不得捏造**假數據、假客戶、假見證當成真實證據。沒有真實數字就用具體能力描述，不要編造「10,000 用戶」。內部用的風險假設（kill assumption）不進公開頁。

## 1. Value Proposition（canvas → 一句話）

先填 canvas，再壓成一句：

| 欄位 | 問題 |
|---|---|
| For whom | 誰是潛在使用者？（具體角色，不是「所有人」） |
| Unhappy with | 現在的替代方案 / workaround 是什麼？痛在哪？ |
| Our offering | 我們是什麼（產品類別） |
| That provides | 核心 outcome / gain（使用者得到的結果） |
| Unlike | 與替代方案的關鍵差異 |
| Proof | 為什麼可信（能力 / 真實數字 / 範例——不可捏造） |

壓成一句（給 hero subhead）：
> For **[whom]** who **[pain]**, **[product]** is a **[category]** that **[outcome]** — unlike **[alternative]**, **[differentiator]**.

## 2. Amazon Working-Backwards PR/FAQ（從客戶視角倒推）

「Working Backwards」= 在做之前，先寫一篇**未來的新聞稿**，假裝產品已上線、從**客戶**角度寫；逼你先講清楚價值，再回頭設計頁面。這就是 project review 用的那種 PR/FAQ——這裡的目的是**萃取 showcase 文案**。

### Future Press Release（一頁內）
- **Headline** — 客戶導向、outcome 的標題（→ hero headline 候選）。
- **Sub-headline** — 一句「給誰、解決什麼」。
- **Problem para** — 今天的痛 / workaround。
- **Solution para** — 產品如何解決（benefit 語言）。
- **Customer quote** — 一句合理、不誇大的客戶心聲（→ 見證語氣範本；若無真實客戶，標為 *illustrative*）。
- **How to get started** — 最短上手路徑（→ hero / final CTA）。

### FAQ
- **外部 5 問**（客戶會問）：多快上手？要不要工程師？我的資料安全嗎？多少錢 / 有免費試嗎？跟 X 有何不同？→ 答案即 showcase 的 feature/benefit 與 objection-handling。
- **內部 3 問**（團隊誠實面對，不公開）：最大風險假設？哪個證據還不存在？最可能的反對意見？

### 萃取規則
PR headline → hero headline 候選；customer quote → 見證語氣；how-to-get-started → CTA；外部 FAQ → feature cards / 信任段落。

## 3. gstack CEO / Founder Viewpoint（讓定位夠大、夠尖）

> 觀點 **adapted from** Garry Tan 的 gstack CEO review workflow / Builder Ethos（source: https://github.com/garrytan/gstack；本機 `~/projects/gstack/{README,ETHOS}.md`、`plan-ceo-review/`）。用「adapted from / inspired by」措辭，**不要**宣稱 Garry Tan / YC / gstack 背書。公開 showcase 不需印出此 attribution。

用這些 lens 寫 **3–5 句 CEO viewpoint 觀點陳述**，確保 value message 不是 me-too：

1. **Problem truth before feature truth** — 真正的痛是什麼、誰在痛、why now（為何是現在）。
2. **The wedge** — 為什麼這比現有 workaround **明顯更好**（不是「也能做」，是「明顯更好」）。
3. **10-star vision** — 把產品想到「明顯卓越」而非「堪用」會是什麼樣子；這是 hero 該指向的北極星，不是功能清單。
4. **Golden-age gain（gstack ethos）** — AI 讓一個人就能做出過去整個團隊的東西；把「複雜 → 變簡單/快/便宜」翻成使用者 gain（例：days not months、no developers、publish instantly）。
5. **誠實的 bet** — 哪個假設若為假會打死這個定位（內部用，避免公開頁吹過頭）。

把 wedge + 10-star + golden-age gain 濃縮進 hero headline 與 feature 框架，讓 showcase 有 **founder 等級的清晰定位**，而非漂亮詞藻。

## 交付物（建議）

除了改好的公開 showcase 文案，附一份 **positioning brief** 作為團隊對齊與後續文案的 SSOT：

1. **Value proposition** — 一句話 + canvas。
2. **Working-Backwards PR/FAQ** — 一頁 PR + 外部 FAQ（內部 FAQ 可留 brief 內部段）。
3. **CEO viewpoint** — 3–5 句觀點陳述（wedge / 10-star / why-now）。

公開頁只放 benefit 版本；strategic backbone 留在 brief。
