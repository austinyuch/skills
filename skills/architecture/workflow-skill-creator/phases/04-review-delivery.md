# 第四階段：预覽優化与交付

## 階段概述

这是Workflow Skill建立的最后一個階段，目标是對產生的Skill进行品質檢查、迭代優化，并最終交付完整的可用Skill。

## 觸發條件

- 第三階段的Skill文件已全部產生
- 或使用者明确表示要进行品質檢查和優化

## 階段目标

1. 展示產生Skill的關鍵部分供使用者预覽
2. 进行六維度專業品質檢查
3. 根据檢查结果进行迭代優化
4. 最終交付完整、高品質的Workflow Skill

## 具體工作內容

### 4.1 品質预覽

向使用者展示產生Skill的關鍵部分，包括：

#### 预覽內容清單

1. **SKILL.md核心部分**
   - YAML前置內容（name和description）
   - 角色定位和價值主张段落
   - 六階段流程概覽
   - 全域約束列表

2. **流程設計概覽**
   - 六個階段的名称和目标
   - 階段推进邏輯
   - 文件管理結構

3. **評估机制設計**
   - 評估維度列表
   - 評分標準概要
   - 優化策略

4. **交互示例**
   - 1-2個典型對话示例
   - 确认方式示例

### 4.2 專業品質檢查

基於六個維度對產生的Skill进行全面評估：

#### 檢查維度1：完整性檢查 (权重：20%)

**檢查內容**：

- 所有必需文件是否齐全（SKILL.md、6個phase文件、README.md）
- 每個文件的結構是否完整
- 是否有遗漏的章節或內容

**檢查標準**：

- **9-10分（优秀）**：所有必需文件齐全，結構完整，无遗漏
- **7-8分（良好）**：文件齐全，有1-2处小的遗漏
- **5-6分（及格）**：文件基本齐全，有若干遗漏需补充
- **0-4分（不及格）**：缺少必需文件或重要章節

**檢查項目**：

- [ ] SKILL.md包含YAML前置內容
- [ ] SKILL.md包含所有必要章節
- [ ] 6個phase文件都已建立
- [ ] 每個phase文件結構完整
- [ ] README.md已建立
- [ ] reference目錄包含必要文件

#### 檢查維度2：邏輯性檢查 (权重：25%)

**檢查內容**：

- 六階段流程邏輯是否合理
- 階段之间的輸入輸出是否匹配
- 流程控制邏輯是否清晰
- 确认机制是否完善

**檢查標準**：

- **9-10分（优秀）**：流程邏輯严密，階段衔接完美，控制清晰
- **7-8分（良好）**：邏輯基本合理，有1-2处小的不顺畅
- **5-6分（及格）**：邏輯可行但有改进空间
- **0-4分（不及格）**：流程邏輯有明显問題

**檢查項目**：

- [ ] 階段推进顺序合理
- [ ] 每個階段的觸發條件明确
- [ ] 輸入輸出关系清晰
- [ ] 文件狀態判断邏輯正确
- [ ] 确认机制設計合理

#### 檢查維度3：專業性檢查 (权重：25%)

**檢查內容**：

- 领域知识是否准确
- 專業术语使用是否恰當
- 評估維度是否符合领域特征
- 工作方式是否符合领域习惯

**檢查標準**：

- **9-10分（优秀）**：领域知识精准，术语專業，完全符合行業特點
- **7-8分（良好）**：基本專業，有1-2处可優化
- **5-6分（及格）**：專業性一般，需要加強
- **0-4分（不及格）**：專業性不足，有明显错误

**檢查項目**：

- [ ] 领域术语使用准确
- [ ] 評估維度符合领域特征
- [ ] 工作方式描述專業
- [ ] 示例贴近實際場景
- [ ] 參考文件具有專業價值

#### 檢查維度4：可用性檢查 (权重：15%)

**檢查內容**：

- YAML格式是否正确
- 文件路径是否有效
- markdown格式是否規範
- 連結是否正确

**檢查標準**：

- **9-10分（优秀）**：格式完全正确，可直接使用
- **7-8分（良好）**：有1-2处格式小問題
- **5-6分（及格）**：有若干格式問題需修正
- **0-4分（不及格）**：格式問題严重影響使用

**檢查項目**：

- [ ] YAML前置內容格式正确
- [ ] name符合kebab-case規範
- [ ] description长度符合要求（≤1024字符）
- [ ] 所有文件路径有效
- [ ] markdown语法正确
- [ ] 内部連結正确

#### 檢查維度5：创新性檢查 (权重：10%)

**檢查內容**：

- 是否有独特的設計亮点
- 評估机制是否有创新
- 交互方式是否有特色
- 是否提供独特價值

**檢查標準**：

- **9-10分（优秀）**：有明显创新点和独特價值
- **7-8分（良好）**：有一定创新，较为独特
- **5-6分（及格）**：基本遵循標準模式
- **0-4分（不及格）**：完全照搬，无特色

**檢查項目**：

- [ ] 工作流設計有特色
- [ ] 評估維度有针對性
- [ ] 交互方式自然友好
- [ ] 提供独特價值主张

#### 檢查維度6：規範性檢查 (权重：5%)

**檢查內容**：

- 是否符合Agent Skill規範
- 文件結構是否標準
- 命名是否規範
- 文件組織是否合理

**檢查標準**：

- **9-10分（优秀）**：完全符合Agent Skill規範
- **7-8分（良好）**：基本符合，有小的不規範
- **5-6分（及格）**：部分符合，需要調整
- **0-4分（不及格）**：不符合規範

**檢查項目**：

- [ ] 符合Agent Skill目錄結構
- [ ] SKILL.md位置正确
- [ ] phases和reference目錄規範
- [ ] 文件命名符合規範

### 4.3 評分和報告

#### 評分計算

```
综合得分 = 完整性得分×20% + 邏輯性得分×25% + 專業性得分×25% +
          可用性得分×15% + 创新性得分×10% + 規範性得分×5%
```

#### 品質等級

- **≥9.0分**：卓越 - 可立即交付使用
- **8.0-8.9分**：优秀 - 建議做少量優化
- **7.0-7.9分**：良好 - 建議優化后使用
- **<7.0分**：需改进 - 必須優化

### 4.4 迭代優化

#### 優化策略

根据综合得分决定優化策略：

**得分≥9.0**：

- 无需優化，直接进入交付
- 詢问使用者是否满意

**得分8.0-8.9**：

- 建議进行一轮小幅優化
- 主要针對高優先順序問題
- 優化后可直接交付

**得分7.0-7.9**：

- 建議进行1-2轮優化
- 解决高優先順序和中優先順序問題
- 目标：提升到8.5分以上

**得分<7.0**：

- 必須进行2-3轮優化
- 全面解决所有识别的問題
- 目标：至少达到8.0分

#### 優化流程

**第一轮優化**（如果综合得分 < 9.0）：

1. 修复所有高優先順序問題
2. 修复部分中優先順序問題
3. 重新进行品質檢查
4. 更新quality_check.md

**第二轮優化**（如果第一轮后仍 < 8.5）：

1. 修复剩余中優先順序問題
2. 優化部分低優先順序問題
3. 再次品質檢查
4. 更新報告

**第三轮優化**（如果仍需要）：

1. 精細打磨細節
2. 追求完美
3. 最終品質檢查
4. 这是最后一轮

#### 優化循环控制

- **最多3轮優化**
- 每轮優化后重新評分
- 达到8.5分以上即可停止
- 第3轮后无论得分如何都进入交付

### 4.5 最終交付

#### 交付清單檢查

在最終交付前，确认以下清單：

- [ ] 所有必需文件已建立
- [ ] YAML格式正确
- [ ] 所有變量已替换
- [ ] 文件间內容一致
- [ ] markdown格式規範
- [ ] 連結全部有效
- [ ] 品質檢查得分≥8.0
- [ ] 使用者明确满意

#### 交付包內容

最終交付的Skill包含：

```
.opencode/skills/{skill-name}/
├── SKILL.md                    # Skill主文件
├── README.md                   # 使用說明
├── quality_check.md            # 品質檢查報告
├── phases/                     # 階段詳細指南
│   ├── 01-{階段1}.md
│   ├── 02-{階段2}.md
│   ├── 03-{階段3}.md
│   ├── 04-{階段4}.md
│   ├── 05-{階段5}.md
│   └── 06-{階段6}.md
└── reference/                  # 參考文件
    ├── PROMPT_TEMPLATE.md
    ├── VARIABLE_MAPPING.md
    └── {其他參考文件}.md
```

## 重要約束

1. **品質標準嚴格**
   - 必須进行完整的六維度檢查
   - 不能跳过任何檢查項
   - 評分必須客观、有依据

2. **優化必要性判断**
   - 得分<9.0都應建議優化
   - 必須向使用者说明優化的價值
   - 尊重使用者選擇（是否優化）

3. **使用者确认机制**
   - 品質檢查報告必須给使用者查看
   - 優化方案必須获得使用者同意
   - 最終交付必須获得使用者确认

4. **文件完整性**
   - 必須產生quality_check.md
   - 如果进行了優化，必須更新報告
   - 保留優化歷史記錄

## 完成标志

當满足以下條件时，第四階段完成：

1. ✅ 已完成六維度品質檢查
2. ✅ 已產生quality_check.md報告
3. ✅ 综合得分≥8.0（或已完成3轮優化）
4. ✅ 所有高優先順序問題已解决
5. ✅ 使用者明确满意并确认交付

## 自然确认詢问

階段完成时，使用以下方式之一确认：

- "品質檢查得分{X.X}分，Skill已经准备就绪！现在可以开始使用了。需要我提供使用建議吗？"
- "{Skill名称}已完成，所有文件都已優化到最佳狀態。您满意这個成果吗？"
- "经过{N}轮優化，Skill品質达到{等級}水平。准备好交付了吗？"

## 工作流程完成

使用者确认满意后，整個Workflow Skill建立流程圆满完成！產生的Skill可以立即投入使用。
