# 第一階段：需求分析与角色定位

## 階段概述

这是Workflow Skill建立的第一個階段，目标是深度理解使用者需求，明确AI助手的角色定位和核心價值主张。

## 觸發條件

- 使用者提出建立工作流Skill的需求
- 或明确表示要进行需求分析
- 或相关目錄不存在任何文件

## 階段目标

1. 收集完整的基础信息（领域、角色、價值主张、使用者画像、輸入輸出等）
2. 透過深度访谈瞭解行業特殊性和專業要求
3. 產生完整的需求分析文件
4. 获得使用者對需求分析的明确确认

## 具體工作內容

### 1.1 基础信息收集

使用以下问卷引導使用者提供核心信息：

#### 基础定位问卷

**A. 领域与角色**

1. **專業领域**：您希望AI助手服务于哪個领域？
   - [ ] 內容创作（写作、編輯、文案）
   - [ ] 產品管理（需求分析、產品設計）
   - [ ] 技術開發（编程、架構、測試）
   - [ ] 設計创意（UI/UX、平面、品牌）
   - [ ] 資料分析（報告、洞察、建模）
   - [ ] 教育培训（课程、教案、評估）
   - [ ] 营销推广（策略、文案、分析）
   - [ ] 其他：****\_\_\_****

2. **角色名称**：如何称呼这個AI助手？
   - 建議：简洁、專業、易记（如Writer、Designer、Analyzer）
   - 格式要求：kebab-case（如content-writer、data-analyzer）

3. **專業身份**：这個助手的專業身份定位是什么？
   - 示例：高級內容策划师、资深產品經理、全栈開發专家

**B. 價值主张**

4. **核心問題**：要解决的主要痛点是什么？
5. **独特價值**：相比其他助手的独特优势是什么？
6. **成功標準**：使用者用后会有什么显著改善？

**C. 使用者画像**

7. **目标使用者**：主要服务物件是谁？
8. **使用者特征**：他们的專業水平、工作場景、行为特點？
9. **使用場景**：典型的使用情境和频次？

**D. 輸入輸出**

10. **輸入類型**：使用者通常会提供什么样的原始材料？
11. **輸出形式**：最終交付什么類型的成果？
12. **品質標準**：輸出成果的品質要求和評判標準？

### 1.2 深度访谈

基於使用者的回答，进行深度访谈，瞭解：

- **行業特殊性**：该领域有哪些独特的專業要求？
- **工作流程痛点**：現有流程的複雜度和主要困难？
- **交互方式偏好**：期望的沟通风格和互動方式？
- **成功案例參考**：有没有类似的工具或助手作为參考？
- **特殊需求**：是否有特定的約束、規範或標準？

### 1.3 需求文件產生

建立 `.opencode/skill-creator/{skill-name}/requirements.md` 文件，包含以下結構：

## 重要約束

1. **完整性要求**
   - 必須收集所有必要信息才能进入下一階段
   - 如果使用者信息不完整，必須繼續提问直到明确
   - 不能假设或猜测使用者的需求

2. **确认机制**
   - 必須等待使用者的明确认可才能进入下一階段
   - 如果使用者提供反馈，必須修改并再次请求确认
   - 必須繼續反馈-修订循环直到获得明确批准

3. **深度理解**
   - 不仅要收集表面信息，还要理解深层需求
   - 要主動识别潜在的問題和机会
   - 要考虑使用者可能没有明确提到的重要方面

4. **專業性**
   - 使用專業的產品經理思维分析需求
   - 识别需求中的矛盾和不合理之处
   - 提供專業的建議和優化思路

## 交互示例

### 示例1：完整流程

**AI**: "您好！我看到您想建立一個workflow skill。让我们從需求分析开始。首先，请告诉我：

1. 这個AI助手主要服务于哪個專業领域？
2. 您希望如何称呼这個助手？
3. 它要解决的核心問題是什么？"

**使用者**: "我想做一個幫助写技術文件的助手，叫TechWriter，主要解决技術文件写作效率低、品質不稳定的問題。"

**AI**: "很好的想法！让我繼續瞭解更多細節：

1. **目标使用者**：主要是谁在使用？是開發者、技術写作人员，还是產品經理？
2. **輸入材料**：使用者通常会提供什么？是代码、接口文件、產品需求，还是其他？
3. **輸出要求**：最終需要什么样的技術文件？API文件、使用者手册、还是架構文件？"

**使用者**: "主要是開發者使用，他们会提供代码和简單的功能说明，需要產生標準的API文件和使用者使用指南。"

**AI**: "明白了。让我确认几個細節：

1. **專業水平**：使用者是初級開發者还是有经验的？
2. **品質標準**：API文件需要遵循特定的規範吗？比如OpenAPI、JSDoc？
3. **工作場景**：是在項目開發過程中持续產生，还是版本發布前集中编写？"

**使用者**: "主要是中高級開發者，需要遵循OpenAPI 3.0規範，通常是在迭代開發中持续更新。"

**AI**: "太好了！现在我對需求有了清晰的理解。让我產生一份完整的需求分析文件..."

[產生requirements.md文件]

**AI**: "需求分析完成！我已经整理了完整的需求文件，包括：

- 核心定位：面向中高級開發者的API文件產生助手
- 輸入輸出：代码+功能说明 → OpenAPI 3.0規範文件
- 工作方式：迭代式、協作式文件產生

请查看是否准确反映了您的需求？如果有需要調整的地方，请告诉我。确认无误后，我们就可以进入流程設計階段了。"

### 示例2：需求調整

**使用者**: "輸出不只是API文件，还需要包括示例代码和最佳實務说明。"

**AI**: "好的，我理解了。让我更新需求文件，在輸出形式部分增加：

- API文件（OpenAPI 3.0規範）
- 使用示例代码
- 最佳實務说明

这样更完整地反映了輸出要求。我马上更新文件..."

[更新requirements.md]

**AI**: "已更新。现在輸出规格更全面了。还有其他需要調整的地方吗？"

## 完成标志

當满足以下條件时，第一階段完成：

1. ✅ 已建立 `requirements.md` 文件
2. ✅ 文件包含所有必要的8個部分
3. ✅ 使用者明确表示满意當前需求分析
4. ✅ 没有遗漏或模糊的關鍵信息

## 自然确认詢问

階段完成时，使用以下方式之一确认：

- "需求分析完成，我已经深度理解了您的{领域}助手需求。是否可以进入流程設計階段？"
- "需求文件看起来完整了，涵盖了{核心要点}。准备好設計六階段工作流了吗？"
- "我已经清楚地瞭解了{Skill名称}的定位和目标。要不要开始設計具體的工作流程？"
