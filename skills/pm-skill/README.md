# PM Skill 產品經理業務邏輯整理助手

這是一個幫助產品經理整理業務邏輯的 OpenCode Skill，專注於資料實體分析和業務流程梳理。

## Skill 結構

```
pm-skill/
├── SKILL.md                 # 主檔案：核心工作流程和全域約束
├── README.md                # 本檔案：使用說明
└── phases/                  # 階段指南資料夾
    ├── 01-collect.md        # 第一階段：收集
    ├── 02-analyze.md        # 第二階段：分析
    └── 03-document.md       # 第三階段：文件
```

## 適用人群

- **產品經理**：不需要懂技術，只需要描述業務需求
- **業務分析師**：需要深入瞭解系統實作細節
- **專案經理**：需要評估變更影響範圍

## 何時使用這個 Skill

當你需要：

- 瞭解現有系統的業務邏輯
- 分析資料實體之間的關係
- 評估需求變更的影響範圍
- 梳理業務流程和狀態流轉
- 為新功能設計提供資料參考

Agent 會自動識別並使用這個 Skill。

## 三階段工作流

```
收集 (Collect) → 分析 (Analyze) → 文件 (Document)
```

1. **收集** - 全面收集資料實體、Model、表格結構、業務程式碼
2. **分析** - 深入分析實體關係、業務規則、狀態流轉
3. **文件** - 輸出產品經理可理解的業務文件

## 核心特點：資料維度分析

### 什麼是資料維度？

資料維度是指資料的粒度和層級，幫助理解資料的業務含義：

| 維度類型   | 說明                   | 範例                 |
| ---------- | ---------------------- | -------------------- |
| 主實體維度 | 業務核心物件           | 訂單、使用者、商品   |
| 明細維度   | 依附於主實體的詳細資料 | 訂單明細、購物車明細 |
| 關聯維度   | 連接兩個實體的資料     | 使用者-角色關聯      |
| 日誌維度   | 記錄變更歷史           | 狀態變更日誌         |
| 設定維度   | 系統設定參數           | 運費範本、優惠規則   |

### 為什麼維度分析重要？

- **需求評估**：瞭解變更涉及哪些資料
- **影響分析**：明確變更會影響哪些關聯資料
- **報表統計**：確定統計的資料粒度
- **介面理解**：理解資料的層次結構

## 資料獲取方式

### 方式一：讀取 Model 檔案

- 分析 Laravel/PHP Model 的定義
- 理解欄位、關聯關係、存取器等

### 方式二：資料庫 MCP

- 直接查詢資料庫獲取表格結構
- 獲取欄位類型、索引、外鍵等資訊

### 方式三：組合使用

- Model + 資料庫資訊交叉驗證
- 獲取最完整的實體資訊

## 使用範例

### 範例1：整理現有業務

```
使用者：幫我整理訂單模組的業務邏輯

Claude：好的，讓我先收集訂單相關的資料實體...
       [收集階段]
       收集完成，包括訂單主表、訂單明細、狀態日誌...
       [分析階段]
       分析完成，我來產生業務文件...
       [文件階段]
```

### 範例2：評估變更影響

```
使用者：我想在訂單中增加運費險，幫我分析影響

Claude：好的，讓我分析現有訂單結構和相關業務...
       [收集並分析]
       增加運費險會影響：
       - 資料：orders 表格新增欄位
       - 邏輯：訂單建立、金額計算
       - 介面：建立和查詢介面
       詳細分析正在產生文件...
```

### 範例3：理解資料關係

```
使用者：幫我分析使用者、訂單、商品之間的關係

Claude：好的，讓我收集這三個實體的資訊...
       [收集]
       - 使用者(users) - 一對多 - 訂單(orders)
       - 訂單(orders) - 一對多 - 訂單明細(order_items)
       - 訂單明細(order_items) - 多對一 - 商品(products)
       我來產生關係圖...
```

## 輸出文件類型

| 文件類型     | 說明                    |
| ------------ | ----------------------- |
| 實體說明文件 | 資料實體的業務語言說明  |
| 實體關係圖   | ER圖（Mermaid格式）     |
| 業務流程圖   | 流程圖（Mermaid格式）   |
| 狀態流轉圖   | 狀態機圖（Mermaid格式） |
| 變更影響分析 | 需求變更的影響評估      |

## 全域約束

- 使用 ultrathink 模式深度思考
- 使用業務語言而非技術語言
- 注重資料維度分析
- 執行任務前建立 To-dos
- 所有文件輸出到檔案

## 注意事項

1. **需要專案程式碼支援** - 最好有 Model 檔案或資料庫連接
2. **複雜模組分批處理** - 大型業務可分模組逐步整理
3. **輸出需要驗證** - 建議與開發確認分析結果
4. **業務語言為主** - 所有輸出使用產品經理能理解的語言

## 工作風格

PM 採用產品經理友善的互動方式：

- 用業務語言溝通，避免技術術語
- 將程式碼實作翻譯為業務流程
- 主動發現隱藏的業務規則
- 提供可視化的關係圖表
- 為變更提供影響範圍評估

透過這種系統化的業務整理方式，幫助產品經理深入理解系統實作，做出更好的產品決策。
