# 階段三：模板選擇階段詳細指南

## 階段目標

選擇或確認適合的UI模板風格。

## 輸入條件

- 介面規劃檔案（`ui_plan.md`）已完成
- 使用者已確認介面規劃方案

## 模板選擇原則

1. 嚴格參考已选模板的樣式和組件結構，除非使用者有特殊要求，不要進行修改。
2. 必須完整讀取模板檔案，並嚴格按照模板檔案的結構和樣式產生程式碼。如果模板檔案太长，需要分多次讀取。
3. 圖標优先使用FontAwesome开源圖標庫，非必要不使用圖標，避免滥用。
4. 嚴格遵循介面規劃，按照ui_plan.md中的設計實作。
5. 使用模板樣式系統，基於选定模板的樣式和組件。
6. 程式碼結構清晰，HTML/CSS/JS分离清晰，層次分明。
7. 詳細程式碼註解，關鍵邏輯必須有註解說明。
8. 語义化HTML，使用語义化標籤，提升可读性。
9. 回應式設計，支援不同屏幕尺寸。
10. 直接可執行，無需建置工具，浏覽器直接打开即可執行。

## 執行步骤

### 步骤1：展示可用模板

向使用者介绍所有可用的UI模板，包括内置模板和使用者自定義模板。

#### 1.1 列出内置模板

**模板1：at-ui-style（預設推荐）**

**風格特點**：

- 主色调：#E94609（橙红色）
- 設計風格：現代、專業、動感
- 視觉效果：渐變按钮、卡片投影、丰富的動画效果

  - 按钮：渐變背景 + 悬停光泽動画

- 通用B端管理系統
- 需要現代化視觉效果的項目
- 強调品牌色彩的系統
- 需要丰富互動反馈的應用

**參考示例**：`.opencode/ui.html`

#### 1.2 檢查使用者自定義模板

檢查 `templates/custom/` 目錄下是否有使用者添加的自定義模板。

如果有，展示自定義模板：

```
"我还发現您添加了自定義模板：
- [模板名称]：[從README.md讀取的說明]

您也可以選擇使用自定義模板。"
```

### 步骤2：推荐模板

根据需求特點，推荐最合適的模板。

#### 2.1 推荐邏輯

**根据業務場景推荐**：

- **採購/订單管理** → at-ui-style（強调操作，需要明确的視觉反馈）
- **資料分析/报表格** → ant-design-style（專業、資料密集）
- **庫存/物流管理** → element-plus-style（友好、易用）

**根据使用者提及的偏好推荐**：

- 提到"現代"、"動感"、"炫酷" → at-ui-style
- 提到"專業"、"簡洁"、"企业級" → ant-design-style
- 提到"友好"、"清新"、"易用" → element-plus-style

**根据功能複雜度推荐**：

- 功能複雜、互動频繁 → at-ui-style（丰富的互動反馈）
- 資料展示為主 → ant-design-style（專注內容）
- 功能適中 → element-plus-style（平衡）

#### 2.2 推荐话术

```
"根据您的[業務場景/需求特點]，我推荐使用 [模板名称]，
理由是：
1. [理由1]
2. [理由2]
3. [理由3]

當然，您也可以選擇其他模板，或者告诉我您的偏好，
我可以調整推荐。"
```

### 步骤3：使用者確認

#### 3.1 讓使用者選擇模板

```
"请您選擇UI模板風格：
1. at-ui-style（預設）- 現代、動感、丰富
2. ant-design-style - 專業、簡洁、高效
3. element-plus-style - 清新、友好、易用
[4. 自定義模板（如有）]

您選擇哪個模板呢？"
```

#### 3.2 確認定制化需求

如果使用者有定制化要求，詢问具體需求：

```
"好的，使用 [模板名称]。

是否需要定制化調整？例如：
- 修改主色调（預設：[颜色程式碼]）
- 調整組件樣式
- 其他定制需求

如果使用預設樣式即可，我們就繼續下一步。"
```

**定制化調整說明**：

- 主色调修改：簡單，只需替换CSS變量
- 組件樣式調整：中等难度，需要修改具體的CSS規則
- 大幅度定制：建議使用者建立自定義模板

### 步骤4：載入模板

#### 4.1 讀取模板檔案

根据使用者選擇，讀取對應的模板檔案：

```bash
templates/[模板名]/template.html
```

#### 4.2 提取模板的樣式系統

如果template.html太长，需要分多次讀取。必須完整讀取模板檔案，並嚴格按照模板檔案的結構和樣式產生程式碼。

從template.html中提取：

- **CSS樣式規則**：所有的樣式定義
- **組件結構模式**：按钮、表格單、表格格、模態框等的HTML結構
- **JavaScript互動模式**：事件處理、狀態管理的程式碼模式
- **颜色系統**：主色、辅助色、狀態色等

#### 4.3 准备用於程式碼產生

將提取的內容准备好，用於下一階段的程式碼產生：

- 頁面基础模板（HTML結構、CSS樣式、JS框架）
- 組件程式碼片段（按钮、表格單、表格格等）
- 樣式變量（如果需要定制化）

### 步骤5：記錄選擇

建立 `.opencode/ui-generator/{功能名}/template_choice.md` 檔案，記錄模板選擇信息。

**檔案模板**：

```markdown

## 選擇的模板

**模板名称**：[模板名]

**選擇理由**：

- [理由1]
- [理由2]

## 模板特點

**風格**：[風格描述]

**主色调**：[颜色程式碼]

**適用場景**：[場景說明]

## 定制化要求

[如果有定制化要求，記錄在這里]

或

無定制化要求，使用預設樣式。

## 模板檔案路徑

`templates/[模板名]/template.html`

## 模板載入狀態

✅ 模板檔案已讀取
✅ 樣式系統已提取
✅ 組件結構已准备
✅ 准备進入程式碼產生階段

## 使用者確認

- ✅ 使用者已確認模板選擇
- ✅ 定制化要求已確認（如有）
```

## 輸出成果

- 建立 `.opencode/ui-generator/{功能名}/template_choice.md` 檔案
- 模板檔案已讀取並准备好用於程式碼產生

## 完成標準

- ✅ 已選擇具體的模板
- ✅ 模板風格符合需求
- ✅ 定制化要求明确（如有）
- ✅ 模板檔案已載入成功
- ✅ 使用者已確認選擇

## 評估維度

- **選擇合理性（60%）**：模板是否符合場景需求
- **模板品質（40%）**：模板是否完整可用

## 常见問題處理

### Q1：使用者不知道选哪個模板怎么办？

主動展示對比，幫助使用者决策：

```
"讓我幫您對比一下這三個模板的差异：

**視觉風格**：
- AT-UI：現代、動感、色彩鲜明
- Ant Design：專業、簡洁、務實
- Element Plus：清新、友好、柔和

**適合的業務**：
- AT-UI：订單管理、採購管理（操作频繁，需要清晰反馈）
- Ant Design：資料分析、报表格系統（專注內容展示）
- Element Plus：庫存管理、内部工具（平衡功能與易用性）

根据您的[具體業務]，我推荐 [模板名]，您觉得呢？"
```

### Q2：使用者想要多個模板混合怎么办？

說明限制，提供替代方案：

```
"每次產生只能使用一個模板，但我可以：
1. 先用一個模板產生，看效果
2. 如果不满意，可以用另一個模板重新產生
3. 或者，您可以建立一個自定義模板，混合两個模板的特點

您希望采用哪種方式？"
```

### Q3：使用者想要修改模板怎么办？

引導使用者建立自定義模板：

```
"如果您需要大幅度定制模板樣式，建議建立自定義模板：

1. 在 templates/custom/ 目錄下建立新檔案夹
2. 复制現有模板的 template.html
3. 修改樣式和結構
4. 建立 README.md 說明模板特點

建立完成後，我就可以使用您的自定義模板了。

需要我指导您如何建立自定義模板吗？"
```

### Q4：模板檔案损坏或缺失怎么办？

提示錯誤，提供解決方案：

```
"抱歉，[模板名] 的模板檔案似乎不完整或损坏。

可能的解決方案：
1. 選擇其他模板
2. 檢查 templates/[模板名]/template.html 檔案是否存在
3. 如果是内置模板，可能需要重新安装UI Skill

您想選擇其他模板，还是需要幫助排查問題？"
```

完成本階段後，使用以下方式確認推進：

```
"模板選擇完成！您選擇了 [模板名]，[模板特點簡述]。

模板已載入成功，所有樣式和組件都准备好了。

現在可以開始產生程式碼了，预計会產生 [N] 個HTML檔案，
包含完整的 [核心功能清單格]。

准备開始產生程式碼吗？"
```
