Spec-Driven Development

規格驅動開發

SDD is the delivery backbone that turns vague intent into governed, testable, reviewable software change. It gives teams their own inspectable workflow for the coding-agent era.

規格驅動開發(Spec-Driven Development, SDD) 是把模糊需求轉成可治理、可測試、可複核軟體變更的交付主幹。它讓團隊在程式代理人時代擁有自己的可檢討工作流。

Why SDD Needs Its Name

為什麼要替 SDD 正名

It is not paperwork

它不是文件負擔

SDD preserves the chain from business need to acceptance evidence. The point is not more documents; the point is fewer untraceable changes, fewer late surprises, and fewer beautiful artifacts that overstate readiness.

SDD 保留從商業需求到驗收證據的鏈條。重點不是多寫文件，而是減少不可追溯的變更、後期才爆出的整合問題，以及看起來漂亮卻過度宣稱 readiness 的 artifact。

It is team-owned workflow

它是團隊自有工作流

Coding agents are fast, vendor-specific, and partially opaque. SDD gives the team a stable operating model that can survive tool changes and support retrospectives, audits, and continuous improvement.

程式代理人(coding agent) 速度快、各家做法不同，而且部分黑盒。SDD 讓團隊擁有不依賴單一 vendor 的穩定工作模式，可支撐檢討、稽核與持續改善。

Three Role Value Proposition

三個角色的價值主張

CEO

執行長(CEO)

Job: connect product investment to market promise. Pain: fast agent output may create expensive rework. Gain: value, acceptance criteria, and evidence stay connected.

任務：讓產品投入連回市場承諾。痛點：代理人產出很快，但錯誤方向會放大返工成本。獲益：價值、驗收標準與證據保持連線。

CIO

資訊長(CIO)

Job: standardize delivery governance across teams and tools. Pain: every agent behaves differently. Gain: SDD creates a repeatable operating model.

任務：跨團隊與工具建立交付治理。痛點：不同代理人的行為與邊界不同。獲益：SDD 建立可重複的 delivery operating model。

DevSecOps

開發安全營運(DevSecOps)

Job: keep CI/CD fast without losing truth. Pain: mock-heavy tests can look green. Gain: test evidence, review verdict, and demo readiness are separated.

任務：維持 CI/CD 速度，同時不犧牲真實性。痛點：mock-heavy 測試容易看起來綠燈。獲益：測試證據、review verdict 與 demo readiness 分層治理。

Ponytail YAGNI Ladder

Ponytail YAGNI Ladder：最小實作紀律

SDD integrates Ponytail's 6-Rung Ladder (MIT License) as a minimal implementation discipline. Core rule: “Lazy, not negligent” — security, validation, and accessibility are never compromised. Only unnecessary features and over-engineering are trimmed.

SDD 整合 Ponytail 的六級階梯（MIT License）作為最小實作紀律。核心規則：「精簡但不輕忽」——安全、驗證與可及性永遠不在精簡範圍內。精簡的只有不必要的功能與過度設計。

1. Does this need to exist?   → no: skip it (YAGNI)
2. Stdlib does it?            → use it
3. Native platform feature?   → use it
4. Installed dependency?      → use it
5. One line?                  → one line
6. Only then: the minimum that works

Collaboration with Code-Review Skill Family

與 Code-Review Skill 家族的協作

Ladder is the upfront self-check during design and implementation — prevents “unnecessary existence.”

Ladder 是設計與實作階段的前置自我檢視——防止「不必要的存在」。

Code-review family (code-refactoring-advisor, test-quality-reviewer, security-risk-reviewer) is the left-shift detection during review — prevents “bad structure.”

Code-review 家族（code-refactoring-advisor、test-quality-reviewer、security-risk-reviewer）是複核階段的 left-shift 偵測——防止「不良結構」。

When they conflict (ladder says “one line,” refactoring advisor says “extract function”), readability and testability is the final judge — not minimizing line count alone.

兩者衝突時（ladder 說「一行就夠」，refactoring advisor 說「抽出 function」），以可讀性與可測試性為最終判準——不是單純行數最少。

Spec Governance Artifacts

Spec Governance Artifacts 的商業意義

These artifacts are not documentation overhead. They are the operating controls that keep agentic delivery traceable, resumable, auditable, and honest about readiness.

這些 artifact 不是文件負擔，而是讓 agentic delivery 可追溯、可恢復、可稽核，並誠實呈現 readiness 的營運控制面。

The execution-level lifecycle rules live in reference/SPEC_GOVERNANCE_ARTIFACTS_GUIDE.md. This business brief summarizes why those rules matter; SDD phase execution should use the reference and phase files as the operational authority.

Requirement Traceability

需求追溯性與 RTM

Why it matters

為什麼重要

Requirement traceability prevents a common delivery failure: a feature looks complete, but one requirement has no design, no task, no test, or no review evidence. RTM makes the gap visible before release.

需求追溯性(requirement traceability) 避免常見交付失敗：功能看似完成，但某個 requirement 沒有 design、task、test 或 review evidence。RTM 讓缺口在 release 前被看見。

How RTM is used

RTM 如何使用

Use RTM to review requirement → acceptance criteria → design → tasks → tests → review verdict / residual risk. It supports scope control, impact analysis, QA coverage review, release go/no-go, and audit.

用 RTM 檢視 requirement → acceptance criteria → design → tasks → tests → review verdict / residual risk。它支撐 scope control、impact analysis、QA coverage review、release go/no-go 與 audit。

Issue Log to SDD Lifecycle

ISSUE_LOG 到 SDD 的生命週期

1. Hold uncertainty

1. 暫存不確定性

Repeated findings, retro items, known issues, or weak evidence can stay in ISSUE_LOG.md while ownership is unresolved.

repeated finding、retro item、known issue 或 weak evidence 可先留在 ISSUE_LOG.md，直到 owner 清楚。

2. Resolve owner

2. 解析 owner

Fold into an active spec, create a CR for completed baseline, keep as a local lesson, or promote only when repeated root cause and evidence are strong.

可 fold 到 active spec、為 completed baseline 建 CR、留成 local lesson，或在 repeated root cause 與 evidence 足夠時 promoted。

3. Reflect only after SDD

3. SDD 正式化後才反映

RTM and SPECS should reflect formal requirements, tasks, tests, review, or CR state. They should not ingest raw issue-log rows as requirements.

RTM 與 SPECS 只反映正式 requirement、task、test、review 或 CR 狀態；不可直接吞 raw issue-log rows 當需求。

Agile Manifesto

SDD supports working software with evidence, customer collaboration through acceptance criteria, and responding to change through CR overlays, impact triage, and optimization.

SDD 以證據支撐 working software，以驗收標準支撐 customer collaboration，並透過 Change Request、impact triage 與 optimization 支撐 responding to change。

Authority Boundary

權威邊界

SDD does not claim runtime readiness by itself. review.md owns acceptance and readiness verdicts. TESTS.md owns test catalog and evidence pointers. SPECS.md and RTM.md are derived summaries and must not overwrite upstream truth. The Ponytail YAGNI Ladder enforces minimal implementation discipline across Phases 1‑5, preventing over-engineering while preserving safety and correctness.

SDD 不會單獨宣稱 runtime readiness。`review.md` 擁有 acceptance / readiness verdict；`TESTS.md` 擁有測試 catalog 與 evidence pointer；`SPECS.md` 與 `RTM.md` 是 derived summary，不可反向覆寫 upstream truth。Ponytail YAGNI Ladder 貫穿 Phase 1‑5 執行最小實作紀律，在保持安全性與正確性的前提下防止過度設計。
