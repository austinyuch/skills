SDD User Manual

Spec-Driven Development Manual

規格驅動開發操作手冊

This manual explains how to run SDD as an operational workflow: when to start it, what to read, which artifacts to create, when to ask for approval, and how to close without overclaiming readiness.

本手冊說明如何實際執行規格驅動開發(Spec-Driven Development, SDD)：何時啟動、要讀哪些檔案、產出哪些 artifact、何時取得確認，以及如何在不過度宣稱 readiness 的情況下 closeout。

Quick Start

快速開始

1. Enter through Spec Master

1. 先經過 Spec Master

Use spec-master when the request may involve routing, resume, CR, issue log, registry, test catalog, or runtime governance.

若請求可能涉及 routing、resume、Change Request、issue log、registry、test catalog 或 runtime governance，先使用 spec-master。

2. Identify the phase

2. 判斷目前 phase

Read SPECS.md, NEXT_STEPS.md, then the active spec artifacts before deciding whether to continue, reconcile, or start new.

先讀 SPECS.md、NEXT_STEPS.md，再讀 active spec artifact，判斷要續做、協調漂移，或開新 spec。

3. Advance one gate at a time

3. 一次推進一個 gate

Do not skip from intent to code. Move through requirements, design, tasks, implementation, review, and optimization with approval points.

不要從想法直接跳到程式碼。依序推進 requirements、design、tasks、implementation、review、optimization，並在 gate 取得確認。

When to Use SDD

何時使用 SDD

Use SDD for

適合使用 SDD

  • New features or complex fixes.
  • Requirement clarification, design, and task planning.
  • Review rejection, retro findings, known issues, or tech debt that need lifecycle handling.
  • Manual, project review, dashboard, auth, E2E, or live demo risk planning.
  • Repo-owned global skill behavior changes.
  • 新功能或複雜修復。
  • 需求梳理、架構設計與任務規劃。
  • 需要生命週期處理的 review rejection、retro finding、known issue 或技術債。
  • manual、project review、dashboard、auth、E2E 或 live demo 風險規劃。
  • repo-owned global skill 行為變更。

Route elsewhere when

下列情境應交給其他 skill

  • Only cleaning TESTS.md: use test-registry-manager.
  • Only refreshing SPECS.md: use spec-registry-manager.
  • Only allocating local runtime: use local-infra-registry-governance.
  • A trivial unit-test fix does not need full SDD unless it reveals lifecycle risk.
  • 只清理 TESTS.md：使用 test-registry-manager。
  • 只刷新 SPECS.md：使用 spec-registry-manager。
  • 只分配 local runtime：使用 local-infra-registry-governance。
  • 單純修一個小型 unit test 不必啟動完整 SDD，除非它揭露生命週期風險。

Safety Guard: Lazy, Not Negligent

安全守護：精簡，非輕率

The ladder eliminates unnecessary code, dependencies, and abstractions. It does NOT eliminate error handling, input validation, security controls, logging, or test coverage. What the ladder removes is speculative generalisation, unused extensibility, and premature abstraction — the three largest drivers of accidental complexity.

Ladder 消除的是不必要的程式碼、依賴與抽象，不消除錯誤處理、輸入驗證、安全控制、日誌或測試覆蓋。Ladder 移除的是speculative generalisation、unused extensibility 與premature abstraction——這三者是多餘複雜度的最大來源。

Collaboration with code-review Skill Family

與 code-review 家族的協作

Ladder (Design / Implementation)

Ladder（設計 / 實作階段）

  • Pre-check: prevents "unnecessary existence."
  • Prompts: "Does this need to exist? Can existing tooling do this?"
  • Operates before code is written or dependency is added.
  • 前置自我檢視：防止「不必要的存在」。
  • 提示：「這需要存在嗎？既有工具是否已能做到？」
  • 在寫程式碼或加依賴之前運作。

code-review Family (Review)

code-review 家族（複核階段）

  • Post-check: detects poor structure, security gaps, test smells.
  • Includes code-refactoring-advisor, test-quality-reviewer, security-risk-reviewer.
  • Operates on committed code and test evidence.
  • 事後偵測：不良結構、安全漏洞、測試壞味道。
  • 包含 code-refactoring-advisor、test-quality-reviewer、security-risk-reviewer。
  • 在已提交的程式碼與測試證據上運作。

Conflict resolution: When the ladder says "one line is enough" but the refactoring advisor says "extract a function," default to readability & testability over line count. A three-line testable function beats a one-line opaque expression.

衝突處理：當 ladder 說「一行就夠」但 refactoring advisor 說「抽出 function」，以可讀性與可測試性為最終判準。三行可測試的 function 優於一行不透明的 expression。

Resume Flow

恢復流程

For a new or resumed session: read SPECS.md, then NEXT_STEPS.md, then the active spec artifacts. If user-stated current facts conflict with old artifacts, enter drift reconciliation before continuing implementation.

新 session 或恢復 session 時：先讀 SPECS.md，再讀 NEXT_STEPS.md，再讀 active spec artifacts。若使用者提供的最新事實與舊 artifact 衝突，先進入 drift reconciliation，不要直接續做 implementation。

Phase Gate Checklist

Phase Gate 檢查清單

  • Phase 1 Gate — Ladder Rung 1 applied: speculative requirements tagged deferred or out-of-scope.
  • Phase 2 Gate — Ladder Rung 2‑4 documented: new dependencies justified with reason why stdlib / platform / existing deps insufficient.
  • Phase 3 Gate — Closeout tasks include ladder-documentation verification for Phase 4/5.
  • Phase 4 Gate — Ladder Rung 5‑6 applied: each implementation is the minimum that works; speculative extensibility removed.
  • Phase 5 Gate — New dependencies/abstractions have ladder records; undocumented items flagged as over-engineering risk.
  • Phase 6 Gate — Repeated over-engineering findings evaluated for process guard or skill update.
  • Phase 1 Gate — Ladder Rung 1 已套用：speculative 需求標記為 deferred 或 out-of-scope。
  • Phase 2 Gate — Ladder Rung 2‑4 已記錄：新依賴已說明為何 stdlib / 平台 / 既有依賴不足以滿足。
  • Phase 3 Gate — Closeout 任務包含 Phase 4/5 的 ladder 文件驗證。
  • Phase 4 Gate — Ladder Rung 5‑6 已套用：每個實作都是最小可用；speculative 擴展預留已移除。

  • Phase 6 Gate — 反覆出現的 over-engineering finding 已評估為流程 guard 或 skill 更新。

  • Ladder: new dependencies/abstractions have ladder documentation in design or task artifacts.

FAQ

常見問題

Can SDD be skipped?

可以跳過 SDD 嗎？

For tiny, local fixes, yes. For cross-module features, governance changes, demo-facing work, or unclear requirements, skipping SDD increases rework and false-green risk.

很小的局部修復可以。跨模組功能、治理變更、demo-facing 工作或需求不清時，跳過 SDD 會增加返工與 false-green 風險。

Does SDD replace code review?

SDD 會取代 code review 嗎？
