Spec Master + Spec-Driven Development

One front door for routing, one backbone for delivery

This is the single merged brief for the two core skills of the Spec Master family: spec-master decides WHO should own a request before any file is edited; spec-driven-development runs the 6-phase lifecycle that turns vague intent into traceable, reviewable change. Kept as two skills on purpose.

Why two skills, not one

Merging the router into the workflow raises intake cost — every ambiguous request would drag a full lifecycle with it.

spec-master — decide ownership

Cheap to enter: classify the request, decide the owner, hand off. No artifacts are written just to find out where work belongs. It also classifies improvement requests: continue an active spec, CR against a completed one, issue-log it, or open a new spec.

spec-driven-development — deliver

A real lifecycle that preserves traceability across phases without pretending to also be a router. One family, two responsibilities: decide ownership first, then deliver.

spec-master — route by authority surface

Route by the authority surface a request touches, not by whichever file was mentioned first.

spec-driven-development — the six phases

1

Requirements

EARS acceptance criteria; YAGNI Rung 1 filters speculative needs.

2

Design & contracts

Bounded contexts, contracts, data models, impact (DDD); Rungs 2–4.

3

Tasks & traceability

Plan closeout for TESTS.md, RTM, SPECS, CR, issue disposition.

4

Implementation

Red → green → refactor with evidence; folder TESTS.md first; Rungs 5–6.

5

Review verdict

review.md owns readiness; code-review family left-shift detection.

6

Optimization

Repeated findings → bounded improvements or promoted process changes.

The one rule that prevents false-green

Evidence flows one way: ISSUE_LOG → spec artifacts & reports → folder-level TESTS.md → workspace rollup → RTM.md → SPECS.md. Derived summaries (SPECS.md, RTM.md) never overwrite the upstream artifacts they summarize, and review.md — not a green dashboard — owns the readiness verdict.

Scrum · DevSecOps · Responsible AI

Scrum

Strengthens boundary management, not ceremonies: PO routes ownership before the backlog hardens; SM holds routing discipline; Developers execute via the right downstream skill.

DevSecOps

Test evidence, review verdict, and demo readiness live on separate authority surfaces, so mock-heavy green CI can't pose as production-ready. Security review is left-shifted into Phase 5.

Responsible AI

Human oversight via phase gates, accountability via separate owners, transparency via authority-surface routing. Aligns with NIST AI RMF, OECD AI Principles, ISO/IEC 42001.
