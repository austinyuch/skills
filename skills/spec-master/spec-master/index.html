Value Proposition

Why it matters

Large agentic projects usually drift because requests are routed inconsistently and the wrong files start acting like authority. Spec Master makes the routing decision explicit before anyone edits a spec, registry, test catalog, or runtime file.

What you get

A packageable governance surface that separates spec authoring, registry sync, test governance, unresolved issue holding, runtime allocation boundaries, and shared guardrails into the right downstream skills.

First Principles

Behavior shift

Agentic AI moves the bottleneck from producing text to coordinating ownership, evidence freshness, and safe file mutation.

Role lens

CEO value is throughput and reuse; CIO value is clear authority; DevSecOps value is evidence integrity. The matching risks are coordination debt, authority drift, and false confidence.

Solution mechanics

Stronger agile, CI/CD, and review practices now need explicit routing, freshness checks, and claim boundaries. Spec Master provides that by splitting routing, registries, tests, issues, runtime, and shared guardrails into separate authorities.

Team-owned workflow

Coding agents change quickly, vendors make different orchestration choices, and many decisions remain partly black-box. Spec Master gives the team an inspectable workflow, evidence trail, and retrospective surface so process improvement belongs to the team.

What You Lose If You Merge These

Router + authoring

If `spec-master` and `spec-driven-development` are merged, intake cost rises because every ambiguous request drags a full workflow with it.

Registry + test evidence

If `spec-registry-manager` and `test-registry-manager` are merged, stale evidence is harder to spot because registry wording and test evidence share one authority model.

Issue log + runtime

If `issue-log-manager` and `local-infra-registry-governance` are merged, planning gaps and operations gaps get mixed together, which hides the real owner.

Routing Model

1

Identify the surface

Decide whether the request is spec authoring, registry maintenance, issue holding, runtime allocation, or shared governance.

2

Route by authority

Send the work to the skill that owns that surface instead of letting one file pretend to be all of them.

3

Keep evidence honest

Let `review.md`, registry snapshots, or runtime evidence speak for themselves, without upgrading them into readiness claims.

4

Hold unresolved work

If ownership is unclear, route to issue-log handling rather than inventing a new spec prematurely.

5

Allocate runtime last

Only when local dev or UAT is required should runtime governance enter the lane.

When To Use

Use it when

A request touches specs, registries, tests, issue holding, runtime allocation, or shared governance and the owner is unclear.

The task is routing work to the right downstream skill before any file is edited.

Do not use it when

The owner is already obvious and the task belongs directly to a downstream skill.

You are trying to produce a readiness verdict instead of a routing decision.

FAQ

Who is this for?

Teams using coding agents on non-trivial repositories, especially when work spans specs, tests, runtime environments, stakeholder documents, and follow-up governance.

Why not just use one large workflow skill?

Because one workflow quickly becomes a blurred authority surface. Spec Master keeps routing separate from authoring, registry sync, test governance, issue holding, and runtime allocation.

Does `spec-master` write specs?

No. It classifies the request and hands branch-spec authoring to `spec-driven-development`.

How does it prevent false confidence?

It caps claims by evidence source, keeps `review.md` as readiness authority, prevents derived-to-derived refresh loops, and routes stale test evidence to `test-registry-manager`.

Evidence And Claim Boundary

This page explains the governance product. It does not claim that a consuming repository is demo-ready, production-ready, or fully integrated. Those claims must come from the consuming repo's own `review.md`, test evidence, runtime registry state, and related authority surfaces.
