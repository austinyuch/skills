Spec Master Manual

Operational routing for spec, test, issue, and runtime work

Use Spec Master when a request touches governance surfaces and the correct owner is not obvious. This manual explains how to route the request, which skill to hand off to, and what to read first before editing files.

route by authority surface read before edit no readiness claims

Quick Start

Use this skill when

The request mentions specs, registry sync, test governance, unresolved improvements, or local runtime allocation and you do not already know the right owner.

Read first

SKILL.md for routing rules, then the downstream skill that owns the surface you identified.

Routing Decision Tree

1

Identify intent

Is the user asking for a new spec, a spec resume, a registry update, test traceability, an unresolved issue, or runtime allocation?

2

Pick the surface

Route by authority surface, not by the file that was mentioned first.

3

Choose the owner

Hand off to the downstream skill that owns that surface.

4

Check evidence

Read the governing artifacts before claiming any status or readiness.

5

Stop at the boundary

Do not turn routing into a new workflow or a readiness verdict.

Common Walkthroughs

A

Create a new spec

Hand off to `spec-driven-development` and start from requirements, then design, then tasks.

B

Update test traceability

Use `test-registry-manager` when `TESTS.md` rows or evidence pointers need refresh.

C

Record a repeated gap

Use `issue-log-manager` when ownership is unclear and the item should be held before a new spec is opened.

Output Interpretation

What this skill returns

A routing decision, a recommended downstream owner, and a conservative explanation of why the request belongs there.

What it does not return

It does not return a readiness verdict, a runtime claim, or a full spec artifact.

Claim boundary: spec-master only routes. It does not declare a product ready, a test catalog complete, or a runtime allocation valid by itself.

FAQ

When should I use spec-master?

When the request touches spec, registry, test, issue, runtime, or shared governance surfaces and the right owner is not obvious.

Does it write specs itself?

No. It routes to `spec-driven-development` for actual branch-spec authoring.

Can it decide readiness?

No. Readiness stays with `review.md` and related authoritative evidence.

Responsible AI Operator Checks

Before routing

Check whether the request needs human owner resolution, touches sensitive runtime data, or could turn an unresolved issue into a premature spec.

After routing

Record the downstream owner and the reason. Do not claim readiness, compliance, fairness, or safety from routing alone.

Evidence And Claim Boundary

This manual explains how to route work and which downstream skill to use. It does not claim that a repo is production-ready, demo-ready, or fully integrated.
